# Gatherlink v0.9.4 Release Notes

Date: 2026-05-29

V0.9.4 is the public-readiness and installability release after the v0.9.3
adaptive-performance baseline. The release focuses on packaging, first-run
setup, secure local operator APIs, config compatibility, public docs, and
support intake. The scope now also includes a focused performance follow-up for
endpoint overhead and WireGuard-over-Gatherlink behavior.

## Scope

- Debian-oriented release artifact planning now includes a `.deb` package with
  CLI entry points, Rust helper binary placement, examples, docs, and inert
  maintainer scripts. Package install must not auto-start tunnels or mutate
  network policy.
- The Debian package installs Gatherlink and Python runtime dependencies under
  `/usr/lib/gatherlink/python`, avoiding collisions with Debian-owned Python
  packages such as `python3-pygments`.
- `gatherlink helpers wireguard-setup` now gives a fuller first-run handoff:
  validate, doctor, plan, start, status, monitor, operator traffic check, and
  explicit cleanup.
- The local status HTTP helper requires API-key authentication. Keys can be
  provided as plaintext input, file input, or stored `sha256:` hashes; runtime
  state and responses avoid exposing plaintext secrets.
- Config migration has a Pydantic-backed chained transform registry with dry-run
  and write modes. The current production schema remains v1; tests prove chained
  upgrade and downgrade behavior with explicit future-version fixtures.
- Public website copy now has a static-only documentation scope and Cloudflare
  Pages notes that are separate from runtime behavior.
- GitHub issue intake remains deliberately small. The consolidated report form
  covers bugs, docs issues, performance results, regressions, scheduler
  comparisons, and lab or VM reports.
- Documentation maintenance now has explicit rules and tests for keeping
  durable docs free of stale version/status claims and for using clickable
  Markdown links in directory README indexes.
- Focused performance follow-up reduced one endpoint hot-path copy, keeps
  TCP-biased core services on the VM-proven normal drain batch, lets
  TCP-biased coordinated scheduling promote only when ordered aggregation is
  proven safe, and separates protected-service path choice from bulk-service
  capacity choice. Split-WireGuard VM proof improved the fiber+5G mixed row
  while the high-BDP satellite/mobile row remains conservative. Ordered
  multipath now avoids equal-path starvation after idle gaps, but benchmark
  evidence still keeps one opaque TCP-heavy WireGuard flow on the conservative
  default rather than automatic packet-level striping.
- Scheduler diagnostics now expose the evidence Python needs before it promotes
  opaque TCP-like services: per-service/per-path outcome rows, receiver
  blocking pressure, in-flight and earliest-delivery facts, ACK/control-return
  quality, and a promotion trace explaining why coordinated adaptive held,
  switched, or rejected a scheduler candidate.

## Breaking Changes

None for the v0.9.4 scope.

## Known Limits

- Debian remains the tested backend for v0.9.4.
- The local REST/status API is protected by API keys but is still not a public
  WAN management API.
- Static crypto remains lab/manual fallback material.
- Cloudflare website notes are static-hosting notes only; Gatherlink does not
  provide hosted accounts or a hosted control plane.
- Relay performance optimization is future-pipeline work unless a later
  roadmap explicitly promotes it.
- High-BDP opaque TCP over WireGuard is safer, not solved. The current
  conservative policy avoids unsafe ordered aggregation; future work should
  improve ordered multipath before making it automatic for satellite/mobile TCP
  profiles.
