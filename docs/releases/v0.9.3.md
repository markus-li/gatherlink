# Gatherlink v0.9.3 Release Notes

Status: draft, updated as work lands on `dev`.

Gatherlink v0.9.3 is the active adaptive-performance and real-world operation
release after the v0.9.2 baseline. These notes are intentionally maintained
during development, not only at tag time, so operator-facing changes,
compatibility notes, and release-gate evidence do not get reconstructed from
memory later.

## Operator Impact Summary

- No breaking user config changes have landed so far.
- New autonomous live-rekey startup flags are optional. Existing authenticated
  configs continue to execute packet traffic without them.
- Older authenticated security blocks that lack v0.9.3 keyless session metadata
  remain packet-execution compatible, but they cannot originate autonomous live
  rekey. Runtime planning emits a warning so operators know why live rekey is
  unavailable.
- Static security mode remains lab/manual only and should continue to produce
  loud warnings.
- Plaintext/no-security mode remains local-lab/debug only and warning-heavy.

## Breaking Changes

None so far.

If a future v0.9.3 slice changes config shape, CLI behavior, service registry
state, helper contracts, diagnostics schemas, or runtime JSON in a way that can
break existing scripts or configs, record it here before the slice is merged.

## Compatibility And Migration Notes

### Authenticated Live Rekey Inputs

Autonomous authenticated rekey now has an explicit foreground-runner path:

```bash
gatherlink run start node-a.json \
  --name core.node-a \
  --scheduler-reapply-interval 5 \
  --rekey-local-identity state/node-a.identity.json \
  --rekey-peer-identity state/node-b.public.json \
  --rekey-topology state/topology.signed.json \
  --rekey-trust-root state/trust-root.public.json
```

All four `--rekey-*` arguments are required together. This is intentional:
packet execution can use the compiled authenticated security block in config,
but autonomous rekey needs Python to verify the local identity, expected peer,
signed topology, and trust root before it sends or accepts rekey control
payloads.

Configs generated by older authenticated provisioning commands can still run,
but they may need regeneration with current Noise provisioning commands before
they can participate in autonomous live rekey. The required additional fields
are keyless session metadata such as local/peer node ids, topology generation,
role, creation/expiry time, and rekey thresholds. Secret key material remains
redacted in runtime/config introspection.

## What Changed Since v0.9.2

- Authenticated-session runtime metadata is preserved from Noise provisioning
  into runtime state so Python can reconstruct session policy without exposing
  traffic keys.
- The reserved auth/crypto service has Python-owned scheduling and dispatch.
  Rust forwards reserved-service bytes and executes compiled AEAD facts; it
  does not decode Noise, choose rekey policy, or interpret topology.
- `LiveRekeyCoordinator` handles duplicate-start suppression, peer
  initiation/response/reject messages, bounded reject payloads, and validated
  replacement-session hot apply.
- `gatherlink run service` and process-managed `gatherlink run start` accept
  explicit live-rekey identity/topology/trust-root inputs.
- Service monitor exposes compact auth/rekey facts without exposing Noise
  payload bytes or keys.
- Hyper-V live-rekey acceptance now provisions temporary Noise-authenticated
  material, starts both nodes with explicit rekey identity/topology/trust-root
  inputs, triggers a single-initiator live rekey while UDP traffic is moving,
  verifies `rekey.started` and `rekey.succeeded` diagnostics on both nodes, and
  proves post-rekey packet delivery.
- Service-budget status is visible through the runner and monitor so
  Python-owned QoS decisions can be explained from structured facts.
- User documentation now includes a minimum Debian setup path that installs
  Python dependencies, installs Gatherlink in editable mode, builds the
  Rust/PyO3 dataplane binding with `maturin`, and runs first smoke checks.
- `gatherlink helpers wireguard-setup` adds an installer-style setup wizard and
  non-interactive generator for multipath WireGuard-over-Gatherlink configs,
  WireGuard skeletons, and split-profile traffic planning.
- The WireGuard setup wizard now uses Typer prompts with Rich panels and tables
  for clearer navigation, generated-file summaries, warnings, and next steps.
- `tools/export_observed_status_profile.py` exports a repeatable benchmark
  profile draft from real Gatherlink service status snapshots, preserving
  observed path counters, latency, jitter, loss, MTU, and pressure.
- `tools/hyperv/run_gatherlink_onehop_speed.sh` can now run direct path-level
  competing UDP traffic beside Gatherlink traffic with `--competing-rate`,
  recording both Gatherlink and competing send/sink results in the benchmark
  report.

## Known Limits

- Autonomous live rekey is VM-proven for the documented single-initiator path;
  simultaneous/colliding rekey initiations are not yet claimed.
- Benchmark-driven scheduler tuning remains active v0.9.3 work.
- Explicit TCP-helper correctness and capped throughput are VM-proven, but this
  is still a helper-path proof rather than a claim that explicit TCP helpers
  beat every WireGuard-over-Gatherlink profile.
- Ordered multipath is preserved as an explicit specialist/research policy, not
  the automatic default for TCP-like UDP services.
- V0.9.3 does not add GUI, hosted accounts, hosted control plane, broad
  non-Debian platform support, kernel/XDP dataplane, dynamic mesh product, or
  hidden reliability for ordinary UDP payloads.
