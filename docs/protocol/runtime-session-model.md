# Runtime Session Model

## Purpose

This document defines how Gatherlink should think about nodes, peer sessions,
services, paths, and relay layers at runtime.

## Node identities

A node identity is the cryptographic and policy identity used for authenticated
Gatherlink relationships.

A single host may run more than one node identity. Prefer separate processes for
separate node identities, with coherent shared configuration generated by
Python. This keeps each node's keys, sessions, helper state, and diagnostics
easy to reason about. A host may still act as both a sink for one relationship
and a source toward multiple destinations.

## Peer sessions

A peer session is always node-to-node.

Relays are layered around peer sessions. A relay hop does not turn the relay
into the endpoint peer for the inner traffic unless that relay is also the
configured endpoint for that service.

Normal multipath uses one peer crypto session across many paths. A path is a
carrier or route option for an already authenticated peer session; it is not a
separate cryptographic identity by itself.

Create separate crypto sessions only for separate trust domains, such as:

- relay-hop sessions
- endpoint/site-gateway sessions
- bootstrap or enrollment paths
- tenant-isolated paths
- intentionally isolated helper/service relationships

## Receiver indexes

`receiver_index` is the public opaque lookup value on encrypted packets.

It must be:

- random or otherwise opaque
- unique among the local node's active receive sessions and key phases
- rotated when a receive session or key phase is replaced
- mapped locally to authenticated peer/session/key state

It is not a stable node identifier, route identifier, service identifier, or
operator-facing name.

## Services and paths

One peer session may carry many services and many paths.

A service may be active across multiple peer sessions when policy allows it.
Examples include a control service reachable over more than one authenticated
relationship, or a service deliberately exposed through multiple exits. Routing
and exit choice must come from authenticated configuration/control context, not
from plaintext packet labels.

`service_id` is local runtime context. Python assigns or compiles it from the
human-facing config and control state. Rust treats it as a compact execution
value after validation.

`path_id` identifies a local path within the session/policy context. It is used
for scheduling, diagnostics, dedupe/reorder context, and path lifecycle, not for
plaintext trust decisions.

## Duplicates and dedupe

Gatherlink may send duplicate encrypted packets over multiple eligible paths.
Each duplicate is independently encrypted by the sender and replay-checked by
the receiver. The receiver may then dedupe using the decrypted logical
`sequence` plus session/service/path policy.

Dedupe is an optimization and correctness aid for multipath delivery. It must
not be part of authentication.

## V1 compatibility view

Plaintext/lab mode uses the compact v1 logical header because it lacks an
authenticated session context. Secure receive decrypts to compact v2.

When older code or diagnostics need a v1-style view after secure receive, Python
or the control plane may build it from authenticated runtime context:

```text
version     <- session/protocol context
kind        <- compact v2 kind_flags
session_id  <- receiver_index/session mapping
service_id  <- compact v2 service_id
path_id     <- compact v2 path_id
sequence    <- compact v2 sequence
```

New hot-path code should prefer compact v2 after decryption and should not
reintroduce redundant plaintext fields.

`route_id` is explicitly removed. Do not keep it as a runtime compatibility
field. Routing through untrusted peers is represented by outer routing/relay-hop
headers and authenticated relay session state. Endpoint service/exit decisions
happen only after endpoint decrypt from authenticated service/control context.
