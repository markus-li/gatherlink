# Hyper-V Current Performance Log

This file is the current fast-reading benchmark view. It keeps only the latest
representative rows for each active benchmark question, with explicit test
dates when rows come from different runs.

Historical experiment details, rejected tuning attempts, and older long-form
evidence live in [`docs/benchmarks/hyperv-performance-history.md`](hyperv-performance-history.md).

## Log Schema Contract

Do not silently remove comparison, pass/fail, or target columns from this file.
The performance log is a current evidence ledger, not prose with occasional
numbers.

Rules for every edited table:

- Keep existing columns unless the replacement table preserves the same
  evidence in clearer columns.
- If a row claims pass, fail, target, guardrail, regression, or release-gate
  meaning, the table must expose its gate basis, pass threshold, and
  performance target either in a short sentence directly above the table or in
  fixed table columns when the values genuinely vary by row.
- When pass and target thresholds are fixed for a table, document them once
  above the table and use compact row-level gate columns with only `fail`,
  `pass`, `target`, or `n/a`.
- If no fair baseline exists yet, keep the comparison columns and write `n/a`.
  The `Reading` cell must name the missing baseline.
- Do not replace `% WG path-set TCP`, `% raw GL total`, or gate columns with
  narrative-only wording.
- Fast comparison tables may omit low-level counters, but detailed tables must
  preserve the columns needed to explain the reading: scheduler/knobs, MTU,
  streams or offered pressure, baseline, delivered result, comparison
  percentage, gate status where applicable, reading, and evidence path.
- Exploratory rows that are not evaluated against a gate should say
  `exploratory` in the relevant gate column rather than looking like release
  evidence.
- Long historical rows can move to
  [`hyperv-performance-history.md`](hyperv-performance-history.md), but the
  current representative row must keep enough columns to recompute the
  conclusion without hunting through chat history.

Canonical threshold names are `pass_threshold` and `performance_target`; see
[`thresholds.md`](thresholds.md). Do not reintroduce older names such as "expected threshold".

## Comparable Layers

Keep benchmark rows separated by layer so speed changes are attributed to the
right component:

- Private LAN: raw VM-to-VM underlay with no WireGuard and no Gatherlink.
- Direct WireGuard: kernel or userspace WireGuard over the same private LAN.
- One-hop WireGuard userspace: direct `wireguard-go` baseline for userland
  WireGuard comparison.
- `wireguard-gotatun-onehop`: optional direct GotaTun baseline; compare it
  beside `wireguard-go`, not as a Gatherlink scheduler result.
- `wireguard-boringtun-onehop`: optional direct Cloudflare BoringTun baseline;
  compare it beside `wireguard-go` and GotaTun, not as a Gatherlink scheduler
  result.
- Gatherlink raw one-hop: UDP payloads through one Gatherlink source and sink.
- Gatherlink raw relay: UDP payloads through an untrusted Gatherlink relay.
- WireGuard over Gatherlink relay: WireGuard traffic carried through the
  Gatherlink relay topology.

CPU and memory are not ruled out unless the matching benchmark row captured
host and VM counters for that run.

## Current WireGuard Over Gatherlink

default. Use `flowlet_adaptive` when an operator explicitly wants a sticky
tunnel and accepts slower reaction to path change. Do not use ordered modes as
the WAN/WireGuard default yet.

  service boundary, even when the traffic inside that tunnel is UDP. Do not
  force `scheduler.traffic_bias=udp` for `run_onehop_wireguard_gatherlink_speed.sh`
  unless the goal is specifically to prove replay-window failure behavior. The
  one-hop WireGuard runner now marks its generated service as `tcp_ordered` by
  default so `coordinated_adaptive` starts from the conservative path policy.

- Clean three-path VM links: start with `coordinated_adaptive` without explicit
  service flowlet pinning for TCP-only scheduler comparison, and use
  `scheduler.traffic_bias=tcp` when the service is TCP-heavy. That bias now
  starts from `single_best_path`, which keeps the best stable path active and
  leaves other paths as drain/probe paths until telemetry justifies a different
  policy. Use explicit flowlet guardrails only when testing the sticky-tunnel
  profile. Eight TCP streams is now the clean tuning baseline; keep higher
  stream counts as load probes rather than assuming they are always better.
- For TCP-heavy WireGuard tests, pass `--scheduler-traffic-bias tcp` to the
  one-hop runner. This exercises the Python coordinator's TCP-biased policy
  selection without creating a new Rust scheduler mode. The 2026-05-24
  post-clock-sync guard reached 926.85 Mbit/s with zero retransmits, while the
  best post-cleanup smoke remains 952.28 Mbit/s.
- Mixed TCP+UDP over the same WireGuard tunnel must be benchmarked with
  `run_onehop_wireguard_gatherlink_speed.sh --mixed`; sequential TCP-then-UDP
  rows are component checks, not normal-traffic proof.
- Fiber+5G facsimile: the current TCP-only shape is `coordinated_adaptive`
  with `traffic_bias=tcp`, no service flowlet pinning, WG MTU 1380, and eight
  TCP streams. Explicit capacity hints matching the known profile produced the
  best row so far at 701.66 Mbit/s; p16 was lower, so more streams are not
  automatically better on this profile. Single-stream fiber is also useful:
  it reaches about 600 Mbit/s through one Gatherlink path versus 722 Mbit/s
  direct userspace WG, which is a real one-path overhead target. Keep WG MTU
  1380 for this case; 1360 was slower and 1400 caused retransmits.
- Starlink/5G facsimiles: pick by traffic goal and by test pressure.
  `capacity_aware` or `coordinated_adaptive` without service flowlet gives the
  best UDP-over-WG aggregation. TCP-over-WG is high-BDP sensitive: p48 was
  under-driving the tunnel, p96 is the current useful stress point, and p128
  was flat while adding retransmits. Keep p96 in future TCP-only rows before
  concluding a scheduler is the limit. Do not use p1 Starlink+5G as a
  Gatherlink tuning target: direct userspace WireGuard over that shaped path is
  already only about 3 Mbit/s for one TCP stream.
- Advanced dual-WireGuard split is now useful on high-BDP mixed profiles. At
  p96, the stable TCP tunnel reaches the same ceiling as TCP-only while the
  fast UDP tunnel stays near 100 Mbit/s with no loss. Keep this advanced mode
  out of the default path, but use it for WG helper tuning and mixed traffic
  acceptance rows.
- Fanout control clock sync is good peer-time telemetry, not a per-path latency
  measurement. Do not feed half-RTT from fanout control frames into scheduler
  path latency. Add a path-pinned control/probe primitive before making
  one-way or per-path latency a scheduler input.
- `arrival_guarded_capacity` is a safe candidate for the coordinator, not a
  new manual default yet. The first fiber+5G and Starlink+5G VM sweeps show it
  preserves raw Gatherlink behavior and can slightly improve Starlink TCP over
  capacity-only scheduling, but it did not beat `coordinated_adaptive`.
- Raw Gatherlink guardrails for shaped profiles should disable service-level
  flowlet pinning. Otherwise a single UDP generator can be intentionally pinned
  to one path and the result stops measuring multipath capacity.
- Path-run bursts are a raw-dataplane tuning knob, not currently a
  WireGuard-over-Gatherlink fix. The 2026-05-23 fiber+5G path-run-32 run
  improved raw delivery to 851.32 Mbit/s of 900, but dropped WG-over-GL TCP/UDP
  sharply.
- WireGuard interface MTU is a real tuning knob for WG-over-GL. Keep 1380 as
  the general starting point, avoid 1420 on normal-MTU Gatherlink paths, and
  test 1280 or 1200 for lossier UDP-first profiles. Use longer confirmation
  runs before treating a short MTU sweep row as stable.
