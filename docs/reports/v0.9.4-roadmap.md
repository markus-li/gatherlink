# V0.9.4 Roadmap

V0.9.4 is the public-readiness and installability release after the v0.9.3
adaptive-performance baseline.

The release should make Gatherlink easier to install, explain, report on, and
operate without widening the core product scope. Performance work remains
benchmark-led. The release now also includes a focused performance follow-up
for endpoint overhead and WireGuard-over-Gatherlink behavior because those
limits directly affect the first-run WireGuard story.

Current performance follow-up status: raw Gatherlink UDP remains clean at the
1 Gbit/s guardrail, and protected opaque WireGuard services now compile to a
service-level `single_best_path` primitive when an ordered node scheduler is
selected without enough safe multipath proof. The same deterministic path pin
is used when coordinated adaptive falls back to `single_best_path` for
TCP-biased protected services, so Python compiles the chosen path id and Rust
does not infer a moving "best" path from noisy runtime tie-breakers. This keeps
explicit ordered and live-reapply commands from causing replay/order loss while
leaving the node path table available for other services, control traffic, and
future probes. True ordered single-flow aggregation is still a future tuning
target rather than a default release claim.

Follow-up primitive status: `inherit` service scheduling can now carry
Python-selected allowed path ids without forcing single-best selection. That
creates the narrow execution hook needed for future ordered protected-service
promotion: Python can select a clean subset of paths, and Rust runs the already
compiled node scheduler only through that subset.

Promotion policy status: Python now promotes protected ordered services only
through a proven subset rather than requiring every otherwise healthy path to
be proven. The promoted subset must include at least two clean paths and enough
capacity to cover observed service demand with headroom; noisy or unproven
healthy paths stay outside the service's allowed set. This keeps the policy in
Python while using Rust's generic `inherit + allowed_path_ids` primitive.
Before that promotion is considered, Python also prefers a single healthy path
that can satisfy observed protected-service demand. Ordered multipath remains
the fallback for cases where one path is not enough and the proven subset is
clean enough to try.

Proof feedback status: Python now evaluates proven real-data safety before the
generic latency-spread fallback, so trusted real traffic can reach the ordered
safety gate. The wider jitter budget is intentionally limited to trusted
real-data latency, not raw clock-sync estimates. Python also consumes existing
per-path control tx/rx frame counters as low-rate probe evidence, which lets
unused paths become eligible without synthetic user payloads. Rust still only
reports counters and executes compiled primitives; it does not own TCP meaning,
promotion policy, or scheduler interpretation. The first VM guardrails kept raw
Gatherlink at the 1 Gbit/s target and correctly kept WireGuard-over-Gatherlink
on `single_best_path` when the live latency window was too noisy to promote.
The latency-quality slice now advertises RTT, clock-error budget, directional
jitter, and p95 over optional control metadata, so peer Python has the
confidence facts needed to distinguish usable one-way latency from noisy
clock-derived estimates without changing compact data packets or Rust policy.
The 2026-05-29 VM pass fixed zero optional latency-stat encoding, restricted
real-data trust to `good` confidence, and corrected path latency proof to use
the selected per-path clock offset rather than a global median offset. The
result stayed conservative: raw Gatherlink remained clean at 999.96 Mbit/s,
while WG-over-GL remained lossless but single-best because the VM timing facts
were still too noisy for ordered promotion. A follow-up pass made rejected
latency candidates non-destructive while accepted samples are fresh, so one bad
probe no longer erases usable proof. The VM row improved to 702.05 Mbit/s with
0.00% loss and still correctly stayed single-best because path-b/c were
clock-proven but not low-jitter payload-proven. Making sink-authoritative
real-data confidence explicit improved the best short row in this slice to
743.07 Mbit/s. Preserving real-data provenance over later clock probes is also
implemented and tested, but its short VM check was not a speed win; keep it as
correctness groundwork. The next clock-safety slice implements the Phase 1
multi-path clock estimator: minimum-delay weighted-median offset consensus,
per-path base RTT, path-spread uncertainty, and path-local drift estimates.
This keeps Python's scheduler facts more honest on high-latency or asymmetric
links without changing compact data headers or moving timing policy into Rust.
The follow-up scheduler guard now keeps TCP-biased/opaque-WireGuard services on
`single_best_path` when clock uncertainty or real-data latency spread is too
large for safe capacity striping. The same noisy timing facts do not block
UDP-biased bulk services from `capacity_aware`, so the lossy-multipath default
remains aggregation-friendly. The benchmark tooling pass also fixed
`udp_pressure` multi-worker sink accounting so idle worker sockets no longer
dilute active receive throughput; the fiber+5G WG-over-GL check now records
621.58 Mbit/s with 8-flow pressure, or 96.2% of its paired iperf UDP receiver
row over 12 seconds, and 596.44 Mbit/s, or 96.4% of paired iperf UDP, over
30 seconds. A deliberately over-offered 920 Mbit/s row confirms the default
opaque-WireGuard service guard remains in force: the service stays on
`single_best_path`, delivers 663.10 Mbit/s by iperf UDP with 12.49% loss, and
the same 8-flow pressure tool reports 677.45 Mbit/s. An explicit ordered
inherit experiment over the same 800/150 Mbit/s asymmetric path set collapsed
to 165.49 Mbit/s with 52.68% loss, so automatic ordered promotion remains
blocked until receiver-feedback scheduling becomes materially better.

Final performance-scheduler evidence status: implemented in the Python
scheduler diagnostic event. The event now includes a per-service path outcome
ledger, receiver blocking-pressure facts, per-path in-flight and earliest
delivery estimates, ACK/control-return quality, and a compact promotion trace
that explains why `coordinated_adaptive` stayed on `single_best_path`, tried an
ordered subset, or rejected striping. Benchmark/report rows still need to stay
disciplined during release work: raw Gatherlink UDP, opaque single-WireGuard
TCP/UDP, advanced dual-WireGuard TCP/UDP split, and mixed traffic should remain
separate so service behavior is not confused with path behavior.

Design discussion before implementation: cooperative tagged packet providers.
This is now planned as a v0.9.5 experiment using the SOCKS5 helper as the first
controlled provider; v0.9.4 should keep the design note and finish the
scheduler evidence loop first.
Gatherlink must continue to treat ordinary WireGuard packets as opaque. The
advanced dual-WireGuard helper already covers the known-class-by-separate-
service case. OS/firewall marks before WireGuard encryption can help operators
route traffic into separate WireGuard profiles, but they do not identify which
encrypted WireGuard packet is TCP, UDP, bulk, or interactive after WireGuard has
produced one opaque UDP datagram. Likewise, Gatherlink-side service labels such
as `tcp_ordered` or `udp_bulk` only describe the whole configured service unless
a cooperating provider supplies per-packet metadata.

The useful future design target is an optional, per-service "enhanced metrics"
mode for cooperative packet providers. A provider could prepend or expose a
fixed-length Gatherlink-private metadata header before the service payload is
handed to Gatherlink. The header would be stripped before egress to the real
service target, never sent to non-Gatherlink peers as application data, and
designed for no-copy or in-place parsing/removal where possible. It should carry
only scheduling/measurement hints, for example:

- provider version and flags
- stable flow or group id for packets that should usually travel together
- traffic class such as ordered, unordered, bulk, latency-sensitive, or control
- split-safety hint for packets that may tolerate striping or duplication
- relative priority and latency sensitivity
- optional packet size/fragmentation preference or "do not coalesce" hint
- optional provider timestamp or sequence for metrics, not reliability

Boundary for this future mode:

- Python owns the semantic meaning, policy, operator opt-in, diagnostics, and
  whether provider hints are trusted enough to change service scheduling
- Rust may parse the fixed metadata cheaply, count/report it, and execute only
  compiled primitives received from Python
- Rust must not grow WireGuard-specific policy, inner-flow inspection, or
  business meaning from the header
- the mode must be optional and per service; today's opaque UDP behavior remains
  the default and must keep working unchanged

## Release Theme

**Debian packaging, secure local API, public docs, first-run setup, and
support-readiness.**

V0.9.4 should leave the project in a state where a Debian-focused user can:

- install Gatherlink from a package or an intentionally documented dev path
- run the setup wizard for common WireGuard-over-Gatherlink shapes
- use a complete local REST API protected by API keys for operator tooling
- understand current security and performance posture from public docs
- report bugs, docs issues, setup issues, and performance results through a
  small issue-template surface
- upgrade or downgrade config files through explicit, tested schema transforms

## Scope

### 1. Debian Package Build And Install Smoke

Status: package artifact build, content audit, release-artifact doctor check,
and WSL `dpkg` install/reinstall/uninstall smoke are implemented and locally
verified. The package installs Gatherlink and Python runtime dependencies under
a private `/usr/lib/gatherlink/python` path so it does not overwrite
Debian-owned Python packages. True downgrade rollback still needs a previous
package artifact when release candidates are cut.

Goal: create a clean Debian package path without silently enabling network
state changes.

Requirements:

- reproducible `.deb` build command from a clean tree
- package content audit proving local `.gatherlink`, VM inventories, private
  paths, secrets, generated reports, and dev-only artifacts are excluded
- install, upgrade, rollback, and uninstall smoke tests on Debian VMs or WSL
  where appropriate
- packaged files include CLI entry points, Rust dataplane binding, examples,
  docs, completions if available, and service unit material if selected
- package installation must not auto-start tunnels, mutate firewall state, or
  enable privileged helpers without explicit operator action

Boundary:

- packaging scripts may install files and register inert service material
- Python still owns config and operator intent
- Debian remains the tested backend; this is not broad Linux packaging support

### 2. First-Run Setup And Helper UX

Status: implemented and locally verified for the WireGuard-over-Gatherlink
setup wizard. Generated handoff includes validate, doctor, plan, start, status,
monitor, operator traffic check, and cleanup commands.

Goal: make the common setup path easy enough for a new user to follow.

Requirements:

- keep the interactive setup wizard as the recommended first-run path
- support the common WireGuard-over-multiple-paths setup
- support the advanced split-WireGuard TCP/UDP model with loud explanation that
  it is an optimization mode, not the default for everyone
- ask operators which interfaces are WAN, LAN, or localhost-only where the
  wizard needs that information
- generate clear next commands for start, status, monitor, test traffic, and
  cleanup
- use helper-owned code for WireGuard, traffic split, DNS, SOCKS5, and TCP
  forwarding behavior

Boundary:

- helpers own helper behavior
- the wizard orchestrates and explains; it must not hide permanent policy in
  lab code or Rust

### 3. Public Website And Documentation Landing

Status: implemented as static website scope notes under `docs/public/`, linked
from the docs index, with Cloudflare Pages notes kept separate from runtime
behavior.

Goal: prepare a small static public site and cleaner GitHub landing path.

Requirements:

- static website scope only
- clear landing page for what Gatherlink is and is not
- links to GitHub releases, quickstart, security policy, benchmarks, helper
  docs, and issue reporting
- visible caveats for Debian-first testing, unaudited security posture, and
  benchmark methodology
- Cloudflare deployment notes separate from runtime behavior

Non-goals:

- hosted accounts
- hosted control plane
- remote management
- telemetry collection

### 4. Issue And Support Intake Polish

Status: implemented. The chooser remains small, and the consolidated report
form covers bugs, docs issues, performance results, regressions, scheduler
comparisons, and lab or VM reports.

Goal: keep public GitHub intake useful without template sprawl.

Requirements:

- keep a small chooser: consolidated report, feature request, setup help,
  security policy, and docs contact link
- the consolidated report must cover bugs, docs issues, performance results,
  regressions, scheduler comparisons, and lab/VM reports
- verify the issue chooser from GitHub after public pushes
- add or update contributor guidance only if it improves the operator path

### 5. Config Migration And Compatibility

Status: implemented and locally verified with Pydantic-backed chained migration
tests for upgrade and supported downgrade behavior. The production config schema
remains v1 until a future roadmap introduces v2.

Goal: make config evolution explicit, reversible where possible, and testable.

Use Pydantic models and the Gatherlink mapping/transform model layer for schema
movement. Migration should be a chain of version-to-version transforms, not a
single ad hoc upgrader. That means:

- `vN -> vN+1` transforms are explicit Pydantic-backed model conversions
- `vN+1 -> vN` transforms are also explicit where data can be represented
  safely in the older schema
- larger jumps are composed through intermediary transforms
- migrations can be explained, tested, and reported field-by-field
- downgrade attempts that would lose required meaning must fail clearly instead
  of inventing unsafe defaults

Requirements:

- versioned config model registry
- `config migrate` or equivalent CLI path for upgrade and supported downgrade
- dry-run mode showing transformed output and warnings
- tests for direct and chained migrations in both directions
- diagnostics for migration start, success, warning, and failure
- docs explaining when downgrade is supported and when it must fail closed

Boundary:

- Python owns schema meaning, migration policy, and operator explanation
- Rust only receives compiled runtime state after validation

### 6. Secure Local REST API

Status: implemented and locally verified for the status HTTP helper. API keys
are required, plaintext keys are not stored in runtime state, non-loopback bind
remains danger-flagged, and the SOCKS5 acceptance proof now exercises the
secured API path.

Goal: provide a complete RESTful operator API with real API-key authentication
for local tools and future UI work.

The API is still not recommended for public WAN exposure. Authentication is a
guardrail for local and private-management use, not permission to expose the
service directly to the internet.

Requirements:

- API-key auth for all non-trivial endpoints
- clear key generation, storage, rotation, and revocation UX
- default bind remains loopback or explicitly private/local only
- loud warnings and explicit danger flag for non-loopback bind
- no secret material in responses, diagnostics, logs, or docs examples
- read endpoints for status, config summary, service list, diagnostics summary,
  benchmark/report pointers, and version/doctor facts
- write endpoints only for safe local operator actions, with expiration or
  explicit enablement where appropriate
- structured error responses with stable codes
- unit tests for auth success/failure, missing key, revoked key, redaction, and
  write-expiry behavior
- smoke test that proves local API usage without relying on lab-only behavior
- docs that say the API is protected but still not WAN-facing

Boundary:

- Python owns API semantics, auth policy, redaction, and operator meaning
- API must call existing orchestration/control paths instead of bypassing them
- Rust does not grow REST semantics or product policy

### 9. Focused Performance Follow-Up

Status: implemented and VM-verified for the focused release slice. The endpoint
fast path, TCP-biased scheduler promotion, protected service path choice,
mixed-service high-jitter policy, and ordered idle-gap executor fix have
code/tests and VM proof rows. The 2026-05-27 single-path fiber+5G VM rerun
reached 651.22 Mbit/s, or 91.2% of the matching direct userspace-WireGuard
TCP-alone baseline. The clean three-path TCP-biased coordinated row reached
896.85 Mbit/s with zero retransmits, or 89.7% of the same-run raw Gatherlink
guardrail. High-BDP opaque TCP over WireGuard remains intentionally
conservative: split-WireGuard keeps UDP clean and protects TCP on the stable
service, while explicit ordered multipath is now correctly using all paths but
still fails the TCP retransmit/throughput gate.

Goal: improve the performance issues that still affect the normal
WireGuard-over-Gatherlink story without muddying ownership boundaries or
turning v0.9.4 into a broad scheduler rewrite.

Work, in priority order:

1. Find and reduce endpoint overhead. One active Gatherlink path should stay as
   close as practical to the equivalent userspace WireGuard and raw Gatherlink
   baselines before blaming multipath scheduling.
2. Improve opaque TCP over WireGuard over Gatherlink. The default behavior must
   protect TCP from bad ordering, queue pressure, and retransmit amplification
   while still using extra paths where evidence says it is safe.
3. Improve ordered multipath safety and usefulness. Ordered modes should use
   latency confidence, receiver pressure, bounded in-flight work, and sender
   pacing better, but they must not hide reliability or become the default until
   benchmark evidence supports that.
4. Improve mixed TCP and UDP split-WireGuard service QoS. Python-owned
   per-service policy should protect the TCP/stable service without starving
   the UDP/fast service, and Rust should only execute compiled queue, pacing,
   priority, and path primitives.
5. Improve high-BDP and volatile profiles such as Starlink plus 5G. The
   scheduler should use credible latency, jitter, loss, queue, and outcome
   signals to avoid overfilling fragile paths while still using available
   capacity.

Requirements:

- keep each benchmark row compared against fair direct WireGuard, userspace
  WireGuard, raw Gatherlink, or previous Gatherlink baselines as appropriate
- prove changes against at least one single-path, one clean multipath, and one
  shaped real-world profile before marking a target complete; this proof is in
  [`docs/benchmarks/hyperv-performance-log.md`](../benchmarks/hyperv-performance-log.md)
- keep policy in Python and execution primitives in Rust
- do not add hidden retransmission, reinjection, or reliability semantics for
  ordinary UDP payloads
- update [`docs/benchmarks/hyperv-performance-log.md`](../benchmarks/hyperv-performance-log.md)
  when benchmark evidence changes
- run focused unit tests and the relevant lab or VM checks for each slice

## Deferred

Do not include these unless a separate roadmap explicitly promotes them:

- dynamic mesh
- broad non-Debian support
- OpenWrt feasibility and production backend work
- kernel/XDP dataplane
- new carrier families
- hosted management
- production relay fabric expansion
- relay performance optimization beyond correctness and basic throughput
- hidden reliability for ordinary UDP payloads
- major ordered-multipath research beyond the focused WireGuard follow-up above

## Release Gates

V0.9.4 can be tagged after release-branch cleanup when:

- the active scope above is implemented or explicitly deferred in this roadmap
- the package path is tested and documented
- the REST API is complete enough for local operator use and protected by API
  keys
- config migration is tested with Pydantic-backed transform chains
- public docs and issue intake are checked on GitHub
- all relevant tests and labs pass
- release notes, living assessment, reports index, and docs freshness checks are
  complete
- the focused performance follow-up is either implemented with benchmark
  evidence or explicitly deferred with reasons in this roadmap and release
  notes
