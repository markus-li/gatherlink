direct: use it to guide code work, test work, and review passes without
re-litigating the architecture.

## V0.9 Target

V0.9 is for personal/lab users and small sites.

Supported platform for v0.9:

- Debian only
- CLI first
- local/lab and small-site deployments
- helpers that are already active or functional
- real two-VM Debian acceptance before calling v0.9 done

V0.9 exclusions followed the project boundaries now captured in
[`docs/architecture/architecture-contract.md`](../architecture/architecture-contract.md); future candidates live in
[`docs/reports/future-roadmap-pipeline.md`](future-roadmap-pipeline.md).

## Implementation Order

### Pass Group 1: Stabilize Acceptance Tests

- Keep the UDP receive retry hardening covered by the Rust path-transport
  tests.
- Run `cargo test --workspace` repeatedly enough to catch obvious timing flakes.
- Run `.venv/bin/pytest -q`.
- Run the WSL acceptance gate.

### Pass Group 2: Debian Compatibility Backend

- Add a platform compatibility layer.
- Keep Debian as the only implemented backend for v0.9.
- Route systemd, journal, `ip`, `ss`, `tc`, `/sys`, capability, and path-layout
  behavior through the backend where practical.
- Do not scatter new OS-specific calls through runtime or helper logic.

### Pass Group 3: Persistence And Secrets

- Formalize Debian paths for config, state, and logs.
- Persist node identity, trust roots, signed bundles, endpoint caches, and
  non-authoritative hints where appropriate.
- Never expose secret key material through status, diagnostics, REST, or JSON
  config output.

### Pass Group 4: Signed Provisioning

- Implement Ed25519 signed topology/provisioning bundles.
- Use canonical JSON for v0.9.
- Keep provisioning out-of-band and friendly to SSH/file-copy workflows.
- Add tamper, stale-generation, and signature-validation tests.

### Pass Group 5: Authenticated Sessions

- Replace static crypto as the v0.9 normal path with a WireGuard-like
  authenticated session setup.
- Keep silent drop on invalid packets.
- Keep receiver indexes opaque.
- Support multiple authenticated source peers sharing the same sink UDP carrier
  port per path. A sink must not require a different public/listen UDP port for
  every source peer.
- Treat this as a v0.9 correctness requirement, not a v0.9.1 polish item. A
  server-like Gatherlink sink must be able to expose one UDP carrier port per
  path, accept packets from more than one authenticated source peer, demux by
  `receiver_index` plus AEAD authentication, and keep UDP source tuples as
  carrier observations rather than peer identity.
- Preserve reply correctness for server-like helpers. When several
  authenticated peers share a sink service, `return_mode:
  "peer-scoped-source"` must give the local UDP target a stable per-peer app
  source socket and route replies back to the authenticated peer/session that
  produced that source.
- Make multi-session sink configs operator-safe. The documented config path
  must show unique local receiver indexes, per-session service mapping, and
  when to use `peer-scoped-source`. Validation or `doctor` should warn when one
  service is mapped to multiple sessions but ordinary fixed return behavior
  would make sink-originated sends ambiguous.
- Add rekey, replay reset, expiry, and golden-vector coverage.
- Leave static crypto as explicit lab/manual mode only.

### Pass Group 6: Secure Relay Sessions

- Implement relay-hop session state from the relay docs.
- Relays are untrusted.
- Relays must not route by plaintext labels.
- Invalid relay packets must not be forwarded and must get no network response.

### Pass Group 6A: Discovery And Remote Status Observability

- Implement discovery, remote status, and monitor-cadence behavior in the
  production core/service path first. Labs may exercise, accelerate, or assert
  this behavior, but must not carry a separate lab-only implementation.
- Keep discovery and remote IPC/status separate.
- Discovery/control metadata is continuous, sparse, and authenticated. It
  advertises stable facts such as service id/name mappings, path labels,
  capacity, MTU, disabled-service assertions, and endpoint assertions for
  verification only.
- Discovery should send at a low baseline cadence and immediately when an
  important fact changes. It should be cheap enough to leave on, and it must not
  stream live status counters by default.
- Remote IPC/status is explicit, louder, temporary, and read-only. It uses the
  remote-status reserved service lane to request status snapshots or a temporary
  status stream from a peer.
- Remote status must auto-expire after the requester stops refreshing it. The
  monitor may keep the request alive while an operator is watching, but the
  peer should return to sparse discovery after the request TTL expires.
- `gatherlink services list` should show local registered services and learned
  remote/discovered services as read-only remote entries. Remote entries come
  from authenticated discovery/control metadata, not from the local service
  registry.
- `gatherlink services monitor` should be able to request temporary remote
  status for a learned remote service and render it as stale/unknown when the
  remote status stream expires or has not arrived yet.
- Monitor-triggered control cadence must be an IPC command supported by the
  production core service, with lab services using the same command shape.
- Remote status must not allow remote mutation. It can expose status snapshots,
  counters, path facts, and diagnostics context, but start/stop/reload/write
  actions stay local or require a separately designed authenticated control
  mechanism.

### Pass Group 7: Experimental REST Helper

- Implement REST as a helper/control-plane sidecar.
- Start it explicitly from CLI.
- Bind to `127.0.0.1` by default.
- Require an explicit danger flag for non-loopback bind.
- Mark it `EXPERIMENTAL`.
- Let read APIs continue while the helper runs.
- Expose write-window metadata now; any future write APIs must expire after one
  hour unless restarted from CLI. Until concrete write endpoints are added,
  mutation requests must fail closed.
- Return structured facts, not secret material.

### Pass Group 8: Helper V0.9 Hardening

Bring currently functional helpers to v0.9 status:

- WireGuard
- SOCKS5 TCP CONNECT
- TCP forward
- DNS
- explicit opt-in time helper

Each helper needs explicit config, managed lifecycle, diagnostics, user docs,
fail-closed defaults, tests, and lab/smoke coverage.

Deferred helpers stay deferred.

### Pass Group 9: Real VM Acceptance Automation

- Add scripts under `tools/vm_acceptance/`.
- Use simple Bash/SSH scripts for v0.9, not Ansible.
- Build the scripts, configs, report format, and docs now.
- Do not ask for VM access yet.
- Do not try to contact real VMs yet.
- The project owner will provide VM access later.

The scripts should install dependencies, sync the repo or package, write
configs, validate configs, start services, send traffic, flap or block a path
when practical, collect diagnostics, stop services, and write a report.

### Pass Group 10: V0.9 Acceptance

Before declaring v0.9 ready:

- `cargo test` passes repeatedly
- `.venv/bin/pytest -q` passes
- helper smoke passes
- WSL acceptance passes
- DNS helper works with direct upstream
- DNS helper tunnel upstream works over Gatherlink and is proven in the VM
  environment
- TCP forward works
- service start/status/monitor/close works
- diagnostics JSONL is parseable and useful
- invalid encrypted packets silent-drop and count locally
- VM acceptance dry run renders and validates configs without contacting VMs
- continuous sparse discovery/control metadata advertises local services and
  path facts without requiring an operator monitor session
- important discovery changes are sent promptly, while unchanged discovery
  facts stay on a low baseline cadence
- remote IPC/status works as an explicit temporary read-only request path and
  auto-expires when the requester stops refreshing it
- labs prove discovery and remote status through the same production-owned
  modules and IPC command shapes used by normal services
- `services list` can distinguish local registry services from learned remote
  services
- `services monitor` can request and display a learned remote service, and show
  stale/unknown when no fresh remote status is available
- multi-source sink acceptance proves at least two source nodes can send to one
  sink using the same sink UDP carrier port per path, with peer/session demux
  based on authenticated receiver-index/session state rather than plaintext
  labels or separate sink ports
- local shared-sink smoke is required but not sufficient for release; the same
  shape must also be proven in the real VM environment
- multi-source sink acceptance must prove sink-side replies return to the
  correct authenticated source when the service uses `peer-scoped-source`

Before tagging `v0.9.0`, the release checklist now shows:

- real two-Debian-VM acceptance passed with project-owner-provided access
- real VM acceptance proved shared sink/multi-source behavior and peer-scoped
  replies
- operator docs show how to build or inspect the multi-session sink config
  without hand-waving the session/service mapping
- config validation and docs call out ambiguous multi-session return behavior
- WireGuard-over-Gatherlink is proven in the real VM environment as
  operator-run WireGuard using Gatherlink-provided UDP service endpoints
- SOCKS5 over Gatherlink is proven in the real VM environment
- relay/routing through an untrusted VM is proven with WireGuard over relayed
  Gatherlink
- VM test reporting records the state of live rekey automation, trust-root UX,
  DoH/full local DNSSEC, WireGuard lifecycle automation, and
  multi-hop relay policy automation
- soak testing ran in a real environment
- user docs are short, accurate, and scenario-based

These checks passed for the v0.9-ready tree as recorded in
[`docs/operations/v0.9-release-checklist.md`](../operations/v0.9-release-checklist.md),
