# V0.9.5 Roadmap

V0.9.5 is the first planned post-v0.9.4 roadmap. Its main candidates are
OpenWrt feasibility and a cooperative tagged-provider performance experiment
using the SOCKS5 helper as the first controlled proving ground.

Do not treat this roadmap as authorization to build broad platform support.
V0.9.4 remains focused on Debian packaging, first-run setup, secure local REST
API, config migration, and public-readiness.

## Candidate Theme

**OpenWrt feasibility, compatibility-backend preparation, and cooperative
service metadata experiments.**

## SOCKS5 Helper Performance Gate

Before SOCKS5 is used as a cooperative tagged-provider test bed, prove the
helper itself is fast enough that it is not hiding Gatherlink scheduler
behavior.

Stage 1: raw-LAN TCP proxy proof.

- run SOCKS5 over raw LAN without Gatherlink in the path
- compare against the normal raw speed baselines used elsewhere in the project,
  including direct `iperf3` TCP, UDP pressure where relevant, and any active
  VM/profile baselines
- test clean LAN, shaped profiles, asymmetric profiles, and high-throughput
  unbounded profiles
- record CPU, memory, socket backlog, and proxy-side buffering observations
- tune the Python helper until it is clearly not the bottleneck, or document why
  it cannot meet the target

Stage 2: SOCKS5 UDP support.

- implement UDP ASSOCIATE support only after TCP proxy throughput is healthy
- test UDP proxy throughput over raw LAN before involving Gatherlink
- compare against direct UDP pressure and `iperf3` UDP rows for the same
  profiles
- include loss, jitter, high-rate, and asymmetric profiles so UDP behavior is
  visible before Gatherlink adds its own scheduling layer

Decision gate:

- if raw SOCKS5 TCP and UDP are close enough to the direct baselines, continue
  with the cooperative tagged-provider experiment in Python first
- if the helper remains too slow after reasonable tuning, document the evidence
  and consider a narrow Rust SOCKS5 proxy datapath while keeping Python in
  charge of configuration, policy, diagnostics, and operator meaning
- do not use Gatherlink-over-SOCKS5 benchmark results to judge tagged-provider
  scheduling until raw SOCKS5 helper throughput has passed this gate

## Cooperative Tagged Provider Experiment

Goal: prove whether explicit per-packet service metadata can improve
TCP-sensitive scheduling without inspecting encrypted payloads or weakening the
default opaque-UDP model.

The first provider should be the SOCKS5 helper, not WireGuard, but only after
the SOCKS5 helper passes the raw helper performance gate above. SOCKS5 gives
Gatherlink a legitimate application boundary where Python already knows the
CONNECT stream, destination, lifetime, and traffic class before it is converted
to Gatherlink-carried UDP stream packets. That makes it a clean test bed for
the optional "enhanced metrics" header discussed in
[`docs/reports/v0.9.4-roadmap.md`](v0.9.4-roadmap.md), while avoiding a
WireGuard fork or packet-inspection shortcut.

Requirements:

- keep pre-change benchmark rows for SOCKS5 TCP over Gatherlink, raw
  Gatherlink, and WireGuard-over-Gatherlink so the experiment has honest
  before/after comparisons
- add an optional per-service enhanced-metrics mode that the SOCKS5 helper can
  enable explicitly
- design a fixed-length Gatherlink-private metadata header that can be parsed
  cheaply and stripped before egress to the real target
- carry enough metadata to group packets that should normally stay together,
  identify ordered versus unordered/bulk behavior, express latency/priority
  hints, and report provider timestamps or sequences for metrics only
- keep the implementation no-copy or in-place where practical; do not add a
  packet-copying helper path just to collect metadata
- expose provider-derived counters and traffic-shape summaries to Python
  diagnostics/status so the scheduler can make better policy decisions
- benchmark TCP through SOCKS5 over Gatherlink before and after the tagged
  mode, including asymmetric and lossy profiles
- keep separate "raw SOCKS5 helper" and "SOCKS5 over Gatherlink" benchmark
  rows so proxy overhead is not confused with Gatherlink scheduler behavior

Boundary:

- ordinary WireGuard packets remain opaque and unchanged
- dual-WireGuard remains the advanced known-class-by-separate-service model
- Python owns the meaning, trust, policy, diagnostics, and opt-in behavior
- Rust may parse/report fixed metadata and execute compiled primitives, but it
  must not grow SOCKS5 semantics, WireGuard semantics, or business policy
- the feature is optional per service; current opaque UDP behavior remains the
  default and must keep working unchanged

## OpenWrt Feasibility Spike

Goal: decide whether OpenWrt belongs in an early post-v0.9 release path without
pretending it is just "Debian but smaller."

Questions to answer:

- What target devices are realistic for CPU, RAM, storage, and crypto
  acceleration?
- Is Python acceptable on target devices, or does OpenWrt require a smaller
  control-plane packaging story?
- Which parts of Gatherlink need an OpenWrt compatibility backend:
  service lifecycle, firewall, policy routing, WireGuard integration,
  DNS integration, package install, logging, and interface discovery?
- Can the Rust dataplane run as-is for the target architectures?
- Does OpenWrt use `nftables` consistently enough for existing traffic-split
  helpers, or is `fw4`/UCI integration required?
- What is the minimum safe operator UX through LuCI, CLI, or config files?

Deliverable:

- a feasibility note with "easy", "medium", "hard", and "do not do yet" areas
- a decision on whether v0.9.5 should build a narrow OpenWrt backend or defer it
- no production OpenWrt backend unless the feasibility result defines tests,
  packaging, diagnostics, and a safe operator path

## Boundary

- Debian remains the tested v0.9.4 backend.
- OpenWrt work must go through a compatibility backend instead of scattering
  platform checks through helpers and runtime code.
- Python owns platform policy and operator explanations.
- Rust should remain portable packet execution and avoid OpenWrt-specific
  product behavior.
