# V0.9.2 Roadmap

Status: release-candidate complete on `dev`. Keep this file as the detailed
implementation and evidence trail for v0.9.2. The compact tag-facing release
document is [`docs/releases/v0.9.2.md`](../releases/v0.9.2.md); future release work belongs in a new
roadmap or in [`docs/reports/future-roadmap-pipeline.md`](future-roadmap-pipeline.md).

Current implementation notes:

- handshake packet shape pre-work is implemented with fixed `mac1`/`mac2`
  trailers and reserved cookie-reply helpers
- carrier adapters now expose lifecycle/drop/send/receive counters, publish
  structured best-effort lifecycle/drop/send/receive diagnostics, and enforce
  configured max datagram size locally
- `least_queue` now uses Python-compiled queue depth, queued bytes, and oldest
  queued age primitives instead of the old in-flight placeholder
- `ordered_multipath` exists as a Python-selected scheduler with Rust-executed
  virtual arrival and in-flight credit primitives; VM relay pressure tests show
  it is now safe around the previous one-path clean ceiling, but not yet a
  throughput win above it. Latest implementation pass: Python now compiles
  ordered in-flight credits from the narrower known TX/RX capacity and
  receiver-pressure ratios, while Rust treats packet/byte credits as a real
  bounded-work gate when another eligible path still has credit. Status:
  implemented, focused unit-tested, and kept as a specialist scheduler until
  future benchmark evidence justifies promoting it for opaque TCP.
- `flowlet_adaptive` and `latency_guarded_capacity` are now exposed as
  Python-owned scheduler policy names; `flowlet_adaptive` compiles service
  flowlet stickiness into existing Rust service primitives, while
  `latency_guarded_capacity` drains obvious latency outliers before Rust
  executes balanced path selection. Status: implemented and unit-tested;
  VM scheduler proof now runs the managed two-node acceptance with live
  scheduler reapply enabled and requires `scheduler.decision` plus
  `config.reapplied` diagnostics.
- scheduler telemetry now carries Python-visible path pressure and jitter facts
  including queue depth, oldest queued age, send failures, receive gaps, reorder
  depth, local drops, stale-control age, loss, and TX/RX jitter. Runtime status
  reapply also merges Rust-exposed path pressure counters into the Python
  scheduler telemetry shape. The service monitor also has a compact queue
  pressure column for these facts. Live reapply now emits structured
  `scheduler.decision` diagnostics with per-path health, score, reasons, and
  compiled primitive facts. The service monitor also has a compact `sch` column
  for the same Python-owned health summary. Status: implemented, unit-tested,
  and VM-proven through the managed acceptance runner with live reapply enabled.
- startup capacity selection now treats explicit config as the initial
  scheduler truth and only uses the non-authoritative cache to fill missing
  values. Live detection and peer telemetry can still update the estimate after
  traffic proves the real path behavior. Status: implemented, unit-tested, and
  VM-smoked in a clean live-reapply WireGuard-over-Gatherlink run.
- TCP-biased coordinated scheduling now treats generic multipath as unsafe for
  one opaque WireGuard/TCP flow. The known-good fallback for TCP-heavy opaque
  tunnels is `single_best_path`; `ordered_multipath_capacity_aware` remains an
  explicit specialist/research scheduler rather than an automatic coordinator
  promotion target. High receiver reorder pressure, jitter/spread, skewed path
  capacity, or missing proof all keep the TCP-biased coordinator on best-path
  behavior. Hot scheduler reapply also skips no-op Rust updates when Python
  recompiles identical scheduler primitives, avoiding avoidable dataplane churn.
  Status: implemented, unit-tested, and VM-smoked in clean, fiber+5G, and
  five-path shaped WireGuard-over-Gatherlink runs. The current evidence says
  this is the safe default, not proof that one ordered WireGuard flow can yet
  aggregate all paths.
- ordered multipath TCP tuning has one additional v0.9.2 pass: Python no longer
  compiles implicit ordered pacing budgets, so Rust bypasses packet-thread
  sleeps unless an operator explicitly configures pacing. Rust keeps a small
  early-arrival smoothing penalty in the cheap ordered scheduler primitive; VM
  sweeps selected multiplier `16` as the current balance. Status: implemented,
  focused Rust/Python scheduler tests passed, VM-smoked on clean TCP-over-WG.
  This remains a specialist/research mode below the `single_best_path` clean
  TCP guardrail, not the production WireGuard default.
- Python now has an adaptive service budget controller fed by Rust service
  counters. Rust still executes only compiled per-service drain primitives;
  Python owns the mixed-service pressure detection, hysteresis, diagnostics,
  and release behavior. Status: implemented and unit-tested. Rust now has a
  byte-budget drain primitive that is bypassed completely unless Python
  compiles a nonzero byte cap. VM guardrails show no raw Gatherlink packet-loss
  regression and a Starlink+5G mixed improvement versus packet-only budgeting,
  but no universal WireGuard-over-Gatherlink win yet. The next service-QoS pass
  should add an explicit runtime outcome signal before making byte-budget
  adaptation more aggressive.
  Tuning note: 128 and 384 stable-packet-equivalent byte caps were both worse
  than the current 256 equivalent on the hard Starlink+5G mixed row, so 256
  remains the current balance. Adaptive tightening exists, but it is deliberately
  sticky and only triggers after extreme sustained bulk dominance; a first less
  conservative adaptive pass hurt the hard Starlink+5G mixed row, so v0.9.2 keeps
  the known-good cap as the normal behavior. This is byte/time service budgeting,
  not a Rust-owned semantic scheduler and not a sleep-based pacing loop. Python
  now also has an optional service outcome DTO and a live service-IPC command
  so helper/benchmark tooling can report protected-service degradation and stop
  the controller from tightening bulk budgets when TCP/application outcomes get
  worse. The runtime still does not invent those outcome facts on its own. The
  dual-WireGuard Hyper-V benchmark runner now records post-run service outcome
  facts in report JSON.
  VM runs on Starlink+5G, Starlink+2x5G, fiber+5G, and the five-path correlated
  Starlink profile prove the DTO/reporting shape and keep benchmark
  interpretation structured. Live closed-loop consumption is now wired through
  the `service-outcome` service IPC command. The Hyper-V TCP outcome probe uses
  per-socket retransmit counters when available and a benchmark-scoped Linux
  TCP `RetransSegs` delta fallback when `ss` is too sparse during short runs.
  The first live Starlink+2x5G mixed pass proved the IPC/runtime feedback loop
  and Python-owned protected-service budget reaction, but it did not materially
  improve TCP-over-WireGuard-over-Gatherlink. That is useful negative evidence:
  bulk-service caps alone are not the missing scheduler; the next TCP/QoS work
  should focus on earlier sender-side pacing or better TCP-outcome prediction,
  still with Python owning policy and Rust executing narrow primitives.
- a Python-owned runtime rekey coordinator can emit start/reject/expired/success
  diagnostics and hot-compile a validated replacement session into runtime
  security facts. The v0.9.2-selected slice is the coordinator, fail-closed
  validation, packet-shape compatibility, and Rust hot-apply contract; a fully
  autonomous peer-to-peer rekey loop with VM traffic proof remains future work.
- `config diff` provides redacted canonical/runtime structural comparison
- `config summary` provides a stable redacted runtime summary for paths,
  services, security sessions, carriers, helpers, and scheduler primitives.
  Status: implemented and unit-tested; lab/VM proof is not required for this
  CLI-only introspection path.
- `doctor` now emits a schema-versioned JSON envelope and reports optional
  operator tools, port/listener facts, and standard-carrier supervision/MTU
  facts without mutating the host
- `doctor` also checks Python/Rust manifest version agreement and tracked-file
  release hygiene without treating local untracked lab state as a failure

V0.9.2 is a stability, cleanliness, and polish release after v0.9.1. It should
not add a new product surface, helper family, carrier family, broad platform
target, or mesh behavior. The goal is to make the existing v0.9/v0.9.1 system
more trustworthy under real use: easier to diagnose, safer to run for longer,
cleaner to operate, and less surprising when paths, carriers, sessions, or
helpers degrade.

## MPTCP And WireGuard Lessons Tracked For V0.9.2

These are roadmap requirements, not claims that the behavior already exists.
They are here so future implementation passes can verify the ideas explicitly
instead of assuming they were handled by earlier performance work. Roadmap
status should be kept explicit: `implemented` means code exists, `unit-tested`
means focused tests passed, and `fully tested` requires the relevant lab/VM
checks listed in the acceptance criteria.

V0.9.2 should carry forward these MPTCP-inspired UDP scheduling lessons:

- flow-sensitive path selection for ordered UDP services, especially
  WireGuard-over-Gatherlink
- flowlet stickiness: keep a busy service/source flow on one path until an idle
  gap, bounded hold time, or sustained path pressure makes moving sensible
- ECF-style predicted completion/arrival estimates using path latency, queue
  pressure, pacing budget, and payload size
- BLEST-style avoidance of paths likely to create avoidable reorder blocking
- live byte/packet credits and in-flight windows compiled by Python and
  executed cheaply by Rust
- bounded pacing to avoid userland burst spikes and VM/socket queue overflow
- TCP-safe ordered multipath promotion only after receiver feedback shows
  several paths have been used cleanly; do not probe by damaging an active TCP
  tunnel first
- path suppression and gradual recovery ramp-up when telemetry proves a path is
  unhealthy or recovering
- explicit acknowledgement/retry only for Gatherlink internal metadata, never
  hidden reliability for ordinary UDP payloads

V0.9.2 should carry forward these WireGuard-inspired operational lessons:

- small stable packet surfaces and no data-header growth for scheduler policy
- silent-drop invalid secure packets with local counters/diagnostics only
- authenticated endpoint-update discipline; unauthenticated packets must not
  mutate peer endpoint state
- rekey overlap and receiver-index clarity while traffic continues moving
- one-path, three-path, and relay comparisons for WireGuard-over-Gatherlink so
  packet striping is not mistaken for useful throughput
- syscall, batching, and datagram-per-second evidence before further hot-path
  tuning claims

## Priority 1: Live Rekey Wiring

Goal:

Turn the existing authenticated session and rekey policy pieces into a running
service behavior.

Current code already has:

- Noise IK-style authenticated session setup
- topology-bound authenticated session planning
- receiver-index generation
- `plan_session_rotation`
- fail-closed decisions for expiry, topology disagreement, and peer mismatch
- Rust replay state that resets when Python supplies replacement session facts

V0.9.2 status:

- implemented and unit-tested: authenticated session planning, receiver-index
  rotation decision logic, fail-closed checks, rekey diagnostics, fixed
  `mac1`/`mac2` handshake packet shape, cookie-reply shape, and compilation of
  validated replacement traffic-key facts for Rust
- implemented in Rust contract: replacement security facts reset the executable
  AEAD/replay state when Python re-applies compiled runtime config
- explicitly not claimed for v0.9.2: autonomous peer-to-peer rekey negotiation,
  live replacement session generation inside `gatherlink run start`, and VM
  proof of an automatic rekey while traffic is moving

V0.9.2 protocol pre-work:

- add and lock the handshake packet compatibility fields for future
  WireGuard-like anti-DoS cookies
- initiation/response packets must carry a fixed `mac1`/`mac2` trailer
- cookie replies must have the reserved fixed packet shape defined in
  [`docs/protocol/security.md`](../protocol/security.md)
- this is format pre-work so Gatherlink does not need a handshake packet
  migration later; full cookie enforcement, source rate limiting, and backoff
  behavior remain future roadmap work

The selected v0.9.2 slice wires this into runtime behavior at the Python
coordinator/Rust hot-apply boundary:

- schedule replacement sessions before expiry or volume thresholds
- overlap old and new receive windows briefly
- hot-apply replacement receiver indexes and AEAD facts without stopping normal
  traffic when safe
- make authenticated endpoint tuple updates explicit: Python may accept a
  changed source tuple only after authenticated traffic proves the peer, and
  must log/diagnose the change without letting unauthenticated traffic mutate
  endpoint state
- fail closed if replacement negotiation is invalid or stale
- emit `rekey.started`, `rekey.succeeded`, `rekey.rejected`, and
  `rekey.expired` diagnostics from the Python runtime coordinator
- keep duplicate/fanout/dedupe behavior covered by the Rust dataplane tests
  that exercise replacement security and replay reset boundaries

Boundary:

- follow the ownership split in [`docs/protocol/security.md`](../protocol/security.md) and
  [`docs/architecture/architecture-contract.md`](../architecture/architecture-contract.md)
- v0.9.2 work is limited to runtime wiring, diagnostics, and proof that rekey
  preserves the existing silent-drop encrypted session behavior
- do not change encrypted data packet headers for handshake anti-DoS
  compatibility; [`docs/protocol/security.md`](../protocol/security.md) owns the handshake packet shape

Acceptance:

- unit tests for success, expiry, stale topology, peer mismatch, replay reset,
  and overlap-window behavior
- diagnostics JSONL proving start/success/reject/expiry events are emitted
- protocol/golden-shape tests proving v0.9.2 handshake packets include the
  `mac1`/`mac2` trailer and cookie-reply framing
- tests must not require full cookie enforcement, source rate limiting, or
  backoff behavior in v0.9.2
- full automatic live-rekey lab/VM proof is deferred until the future
  peer-to-peer rekey loop exists

## Priority 2: Carrier Diagnostics And MTU Hardening

Current code already has:

- Python-owned QUIC and HTTP/3 DATAGRAM adapters
- local byte-preserving carrier smokes
- direct VM carrier smoke evidence on VMs A, B, and C; Traefik UDP-forwarded
  carrier smoke tooling remains optional and runs only where Traefik is
  installed
- stable event codes reserved for carrier lifecycle and datagram facts
- `carrier_max_datagram_size` config surface

V0.9.2 should add:

- lifecycle diagnostics for carrier ready, closed, connect failed, and datagram
  dropped. Status: implemented and unit-tested for direct QUIC/H3 adapters; VM
  direct-carrier proof has passed on all three VMs. Traefik proxy proof is an
  optional environment-dependent check.
- useful sent/received/drop counters without flooding logs. Status: implemented
  and unit-tested for direct adapters.
- clearer handling of max datagram size and effective payload MTU. Status:
  implemented and unit-tested for config/doctor and direct adapter local drops.
- fail-closed diagnostics when QUIC/H3/proxy setup is invalid. Status:
  implemented and unit-tested for direct carrier startup/local-drop failures
  and carrier-supervisor startup failures; proxy availability remains an
  environment check and does not add Gatherlink packet meaning to the proxy.
- proxy-specific diagnostics that still do not give the proxy Gatherlink
  routing, helper, identity, or packet meaning. Status: implemented through the
  carrier-proxy smoke/report path; Traefik rows are optional and skipped unless
  the proxy is installed.
- impaired-network carrier comparison reports for UDP, direct QUIC DATAGRAM,
  HTTP/3 DATAGRAM, and Traefik-forwarded paths. Status: `gatherlink lab
  carrier-compare` now emits stable text/JSON rows for direct UDP, direct QUIC

## Priority 3: Scheduler Telemetry And Fairness

Goal:

Make scheduling decisions more honest and diagnosable before treating adaptive
or queue-aware behavior as production-grade optimization.

Current code already has:

- compiled scheduler policies
- live scheduler hot reapply from status
- path capacity, latency, loss, and MTU primitives
- service priority labels
- deterministic fallback behavior when metrics are missing or malformed

Code-discovered gaps:

- Rust queue primitives now exist in the runtime contract; broader async
  carrier queue telemetry should keep using those same fields rather than
  adding scheduler policy to Rust
- service priority exists, but cross-service fairness and starvation prevention
  need stronger tests and diagnostics
- adaptive policy needs smoothing, decay, confidence, and provenance rules
- ordered single-flow multipath needs real delivery feedback, not only static
  configured path facts, before it can safely exceed the one-path clean ceiling
- normal UDP payloads must remain best-effort; Gatherlink must not add hidden
  retransmission or reinjection semantics for user UDP traffic

V0.9.2 should add:

- an explicit MPTCP/WireGuard lessons checklist for this release so the
  performance work does not become scattered implementation folklore:
  flowlet stickiness for WireGuard-like ordered UDP, ECF/BLEST-inspired
  predicted-arrival scheduling, live credits/windows, bounded pacing, cheap
  queue/reorder/drop telemetry, silent-drop invalid packet posture, explicit
  rekey/endpoint-update discipline, and syscall/batching evidence before more
  hot-path tuning
- Rust queue telemetry for queued packets, queued bytes, oldest queued age, and
  recent drop/backpressure state
- Python smoothing/decay/confidence rules for noisy telemetry. Status:
  implemented and unit-tested for live reapply telemetry; capacity, latency,
  loss, and jitter are smoothed while immediate pressure facts remain direct.
- service-priority fairness checks that prevent low-priority starvation and
  high-priority invisibility. Status: implemented and unit-tested in Python as
  a bounded runner poll order; every listening service receives at least one
  slot and higher-priority services receive a small number of extra slots.
- operator diagnostics explaining why a path was preferred, drained, avoided,
  or marked busy. Status: Python path health/scoring explanations are
  implemented and unit-tested; those facts are now emitted by the live
  scheduler reapply loop as `scheduler.decision` events and rendered in the
  monitor `sch` column. VM proof now runs with live scheduler reapply enabled
  and validates `scheduler.decision` diagnostics.
- live per-path delivery timing telemetry for scheduler use: current, mean,
  variance/jitter, and recent high-percentile approximations where practical.
  Status: implemented and unit-tested in Python control-plane telemetry as
  current, rolling mean, mean-absolute jitter, and nearest-rank p95 fields.
- Rust-edge summarized TX/RX timing facts for scheduler use. Status:
  implemented and unit-tested as cheap aggregate last-TX/RX timestamps and
  inter-packet gap counters on the normal service/path status rows; Python
  still owns all interpretation and scheduler policy.
- per-path pressure telemetry covering queue depth, oldest queued packet age,
  send failures, receive gaps, reorder-buffer depth/age, scheduler in-flight
  packets/bytes, predicted delivery delay, socket buffer capacity, local drops,
  control-metaband inter-frame gaps, and stale control metadata. Status: implemented and unit-tested through
  path-pressure control metadata, local path-stat conversion, Rust scheduler
  runtime snapshots, monitor/scoring facts, and scheduler telemetry parsing.
- scheduler decision diagnostics for the selected mode, per-path scores,
  selected path, suppression/degradation reason, and current credit/window.
  Status: implemented and unit-tested for `scheduler.decision` diagnostics,
  monitor `sch` facts, path health scoring, and compiled credit/window fields.
- scheduler hot-reapply cadence, hysteresis, and provenance rules so noisy or
  stale telemetry cannot make path weights, credits, or suppression flap.
  Status: implemented and unit-tested through the Python coordinator dwell
  packet guard, time guard, confidence windows, bounded decision history, and
  `scheduler.decision` diagnostics; Rust receives only the selected compiled
  primitives.
- `coordinated_adaptive`: Python-level scheduler coordinator that selects a
  concrete policy from telemetry, configured/observed path speeds, reorder
  pressure, queue/loss pressure, and latency spread. Status: implemented and
  unit-tested with packet-volume dwell, a short time dwell guard, confidence
  windows, bounded decision history, diagnostics, and local lab benchmark
  evidence. Rust still receives only the selected compiled primitive policy.
- live-credit `ordered_multipath`: Python adjusts byte/packet credits from
  delivery timing, reorder pressure, drops, and configured capacity; Rust only
  executes the compiled credits and virtual timeline. Receiver path-pressure
  summaries now travel over optional control metadata and feed this credit
  calculation. Status: implemented and unit-tested; fresh VM sweeps show the
  credit path prevents the old ordered-mode collapse but still does not beat
  `coordinated_adaptive` for opaque WireGuard/TCP, so it remains an explicit
  research/specialist policy rather than the default.
- `flowlet_adaptive`: keep a service/source flow on one path while packets are
  close together, then move after idle gap or sustained path pressure. Status:
  implemented and unit-tested as a Python policy that compiles service flowlet
  stickiness and pressure-adjusted weights into existing Rust primitives.
- `latency_guarded_capacity`: prefer capacity only among paths inside a
  Python-selected latency/jitter bound so capacity does not create avoidable
  reordering. Status: implemented and unit-tested as Python-side path draining
  before capacity-oriented primitive compilation.
- Rust-executed per-path pacing budgets compiled by Python to reduce userland
  burst spikes and socket/VM queue overflow. Status: implemented and
  unit-tested as `pacing_budget_bps`; zero bypasses the Rust pacing branch, and
  ordered policies compile capacity-derived pacing budgets for the virtual
  timeline.
- path health scoring in Python (`alive`, `degraded`, `down`) from recent
  send/receive success, drops, latency, jitter, and control metadata freshness.
  Python now also applies those scores during scheduler compilation: degraded
  paths have their compiled weight reduced, while down paths are moved to
  `drain` so normal scheduling avoids them without hiding the path from
  control/probe behavior. Status: implemented and unit-tested for scheduler
  telemetry inputs, live reapply diagnostics, monitor rendering, and compiled
  path suppression; VM proof now covers live reapply diagnostics under
  asymmetric shaping.
- fast path suppression and recovery ramp-up: stop scheduling normal UDP to a
  clearly bad path, continue optional low-rate probes/control where configured,
  and restore credit gradually when the path recovers. Status: initial
  Python-owned suppression is implemented and unit-tested; VM acceptance now
  proves path fail/recover under clean, asymmetric, lossy, and latency profiles.
  Longer soak remains the release gate for subtle ramp-up regressions.
- explicit control-service ack/retry for Gatherlink internal metadata only;
  this must be a reserved/internal service behavior and must not change normal
  UDP payload semantics. Status: remote-status reserved service now tracks
  pending request ids, acknowledged responses, and timed-out requests in Python;
  broader control-metadata ack/retry remains future-only unless a concrete
  internal metadata lane needs it.
- path MTU discovery/refinement and downgrade triggers from safe probes,
  fragmentation symptoms, loss/drops, and carrier max-datagram facts. Status:
  passive interface detection already exists; carrier max datagram ceilings and
  explicit too-large/fragmentation-failed downgrade recommendations are now
  implemented and unit-tested. Generic loss alone intentionally does not trigger
  MTU downgrade.
- shared-sink multi-peer stress coverage so one sink UDP port with several
  authenticated sources remains scheduler- and monitor-safe. Status:
  implemented and unit-tested; local shared-sink smoke passed with two
  authenticated sources sharing one sink carrier port.
- relayed paths use ordinary path telemetry plus relay service diagnostics.
  Status: resolved by design, with no special relay scheduler namespace.
  Relayed traffic is just routed traffic from the endpoint scheduler's point of
  view, like the rest of the internet; relay runners expose counters and
  diagnostics, while endpoints schedule from the same
  latency/loss/queue/path-health facts they use for direct paths.
- clock-safe one-way latency estimator for scheduler use. Status:
  implemented and unit-tested. Python now keeps validated NTP-style
  four-timestamp samples, per-path RTT/offset summaries, confidence/error-budget
  fields, robust median offset summaries, and explicit rejection facts for
  impossible or abruptly jumping samples. The production runner sends sparse
  sync requests from non-sink nodes on the exact path being measured, and the
  sink answers each request on that same path through a narrow Rust
  path-pinned reserved-service transmit primitive. Python folds accepted replies
  into internal clock metadata, derives directional TX/RX one-way estimates from
  the selected peer clock offset, and promotes only sane samples into
  scheduler-visible `clock-synced-one-way` path latency. Real payload traffic
  now also produces sparse Rust-collected tx/rx timing samples keyed by existing
  path id and sequence fields; Python advertises transmit samples over the
  control metaband in the shared clock domain and accepts matched receive
  samples as `data-traffic-one-way` only after the same sanity checks. The Rust
  sampler uses the OS monotonic clock for those sparse data samples, not
  per-process uptime, and can sample a representative sequence inside a
  coalesced batch so batching does not accidentally hide all latency evidence.
  The control metadata payload also carries compact per-path latency
  source/confidence codes, so peer-advertised latency keeps its provenance
  instead of collapsing into a generic `peer` source before the scheduler sees
  it. The local dual-path lab and Hyper-V VM runs now validate the full path by
  showing `data-traffic-one-way` records in service status after real UDP/WG
  traffic. The `reply-rtt-half` source remains the coarse fallback, and service
  monitor output must show whether each path latency value is clock-synced,
  real-data, RTT/2, peer-advertised, or rejected. Rust still receives only
  compact compiled primitives after Python decides the measurement is sane.

  fast services now have separate Python-compiled path policies, path

  not prove priority alone as a reliable fix. Concurrent Starlink/mobile tests
  still show the next missing automatic policy layer is service-level

  enough runtime/path budget to hurt the stable TCP service. The policy
  decision belongs in Python config and scheduler compilation; Rust should only
  execute bounded per-service pacing/queue primitives if this is added.

  quanta into the runner plan; Rust only executes those nonblocking drains. VM

  the knob stays available for explicit budget experiments. The next internal
  service-fairness target should be adaptive byte/time budgeting from Python
  using service counters, not a static packet quantum.
- VPN payload classification decision: we discussed embedding or owning a
  WireGuard-side packet classifier so Gatherlink could identify inner TCP
  streams directly. That is rejected for now because it would turn Gatherlink
  into a VPN implementation and blur helper/core boundaries. Status:
  implemented and focused unit-tested for the config/runtime/scheduler surface:
  Python-owned service `traffic_class` metadata can come from explicit config

  service-traffic summary so `auto` can choose TCP-safe, UDP-capacity, or mixed
  protected/bulk posture without payload inspection. Service budgeting and
  service path allocation use the same centralized Python-owned intent helpers
  instead of duplicating priority/class logic. See
  [`docs/architecture/performance-philosophy.md`](../architecture/performance-philosophy.md).
- Rust service counters are now copied into Python runner state as
  `service_stats`, and Rust also exposes `service_path_stats` for the
  service/path intersections it actually executed. This keeps the next budget
  controller on the correct side of the boundary: Rust reports compact facts,
  Python names paths, interprets service behavior, and decides whether and how
  to recompile service fairness primitives.
- Python now has a dynamic service path allocator for `coordinated_adaptive`.
  It treats configured per-service path sets as starting hints, keeps protected
  services sticky with failover, and lets bulk/fast services expand or shrink to
  the smallest healthy spare path set that covers observed transmit demand with
  headroom. Rust still only executes `single_best_path`,
  `weighted_round_robin`, allowed path ids, and weights; it does not learn
  helper meaning or decide traffic class policy. The allocator now also consumes
  bounded Python-side service outcome feedback for diagnostics, but degraded
  service outcome alone does not steal another service's path while cheap path
  counters remain healthy. Scheduler diagnostics include the latest service path
