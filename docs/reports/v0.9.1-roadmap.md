# V0.9.1 Roadmap

Status: closed historical roadmap. V0.9.1 was tagged from the cleaned release
history and superseded by later release work. Current release status lives in
`docs/project-living-assessment.md`; future work lives in
`docs/reports/future-roadmap-pipeline.md` until a new release roadmap opens.

This roadmap starts after narrow v0.9 has passed real VM acceptance, relay/routing
proof, and operator soak. V0.9.1 should not expand Gatherlink into a broad mesh
product. It should make the v0.9 shape easier to operate, rotate, diagnose, and
prove in small sites.

## V0.9.1 Theme

V0.9 proves that Gatherlink can run as a compact, authenticated, multipath UDP
transport for personal/lab users and small sites.

V0.9.1 should make that safer and smoother:

- rotate sessions without manual interruption
- add standard QUIC DATAGRAM and HTTP/3 DATAGRAM carrier adapters that wrap
  already-formed Gatherlink packets without changing Rust packet semantics
- make trust-root and topology operations easier to inspect and recover
- deepen helper scenarios that real users are likely to run
- turn VM acceptance from one release gate into a repeatable regression harness
- keep deferred/future features as docs until they have a real owner and tests

## Deferred From V0.9

These items were intentionally non-blocking for the v0.9 tag after the real VM DNS
tunnel proof and one-hour operator soak passed. They are recorded here so v0.9 can
stay narrow without losing the next useful hardening work:

- live rekey and receiver-index rotation automation
- richer trust-root lifecycle UX and signed-topology recovery commands
- DNS-over-HTTPS helper upstreams, implemented through dnspython's HTTPS
  transport without changing the Gatherlink core transport
- full local DNSSEC validation if upstream AD-bit policy is not enough
- WireGuard lifecycle automation if users want Gatherlink to supervise more of
  the surrounding WireGuard process flow
- operator-safe lab bundle generation for repeatable multi-node setups,
  including WireGuard-over-Gatherlink demos, without silently mutating host
  networking
- multi-hop relay policy automation
- fixes discovered during VM acceptance or soak runs that are not v0.9 blockers

## Release Scope

V0.9.1 exclusions followed the project boundaries now captured in
`docs/architecture/architecture-contract.md`; future candidates live in
`docs/reports/future-roadmap-pipeline.md`.

## Release Gates

Before declaring v0.9.1:

- all v0.9 gates still pass
- real two-Debian-VM acceptance is automated enough to rerun after changes
- a longer soak run has useful diagnostics and no unexplained service leaks
- docs describe exactly what is implemented, deferred, or VM-gated
- no deferred helper exists as runtime code unless it has real behavior,
  ownership, tests, and docs
- UDP, direct QUIC DATAGRAM, and HTTP/3 DATAGRAM carrier behavior are covered by
  tests/labs. UDP remains the Rust runner's baseline carrier. QUIC DATAGRAM and
  HTTP/3 DATAGRAM are Python-owned standard-protocol adapters that carry opaque
  Gatherlink packets between local Rust UDP path sockets and remote carrier
  peers. Directly asking the Rust UDP runner to execute a non-UDP carrier still
  fails closed.

## Priority 1: VM Acceptance Becomes Regression Infrastructure

Goal:

Turn the first two-VM proof into a stable regression harness.

Needed:

- keep the simple Bash/SSH harness
- add a clear report schema for pass/fail, skipped, not configured, and
  deferred checks
- preserve commands, sanitized configs, monitor output, diagnostics JSONL, and
  service lifecycle logs
- include WireGuard-over-Gatherlink and SOCKS5-over-Gatherlink scenarios when
  the VM inventory supports them
- record path-flap/degradation/recovery behavior
- keep VM secrets and hostnames out of committed files

Boundary:

- VM tooling lives under `tools/vm_acceptance/` or lab docs
- VM tooling may call SSH, SCP, shell, and platform setup scripts
- core runtime must not learn VM-specific assumptions

## Priority 2: Live Rekey And Receiver-Index Rotation

Goal:

Move from config-compatible authenticated session material to managed live
rotation for already-running services.

Needed:

- schedule rekey before expiry. First policy helper implemented in Python as
  `plan_session_rotation`.
- overlap old/new receive windows briefly. First helper reports
  `overlap_until`; service hot-reapply wiring still needs the full receive
  window swap.
- rotate receiver indexes without a public negotiation oracle. First helper
  always chooses a different opaque next receiver index.
- reset replay windows per new receive session. Rust already resets replay
  state when Python compiles a replacement receiver index/key set; live wiring
  remains the next step.
- emit diagnostics for rekey start, success, rejection, expiry, and fallback.
  Stable event codes and factory are now present.
- fail closed if peers disagree on topology generation or identity. First
  helper returns fail-closed decisions for both.

Boundary:

- follow `docs/protocol/security.md` and
  `docs/architecture/architecture-contract.md`

## Priority 3: Trust-Root And Topology UX

Goal:

Make signed topology/trust-root operations less fragile for small-site users.

Needed:

- safer trust-root list/export/import output
- signed identity rotation flow
- explicit stale-generation and revoked-generation diagnostics
- bundle diff/check command that explains what will change before install.
  First slice implemented as `gatherlink secrets topology-diff CURRENT
  CANDIDATE --trust-root ROOT`, including generation, node, service, revoked,
  and issuer-change facts.
- rollback guidance for bad topology bundles
- state-audit output that is useful in VM reports

Boundary:

- signed artifacts stay canonical and inspectable
- no secret key material in operator output
- no hidden enrollment service for v0.9.1 unless explicitly designed later

## Priority 4: DNS Helper Deepening

Goal:

Make the DNS helper trustworthy enough for real small-site use without turning
it into an enterprise DNS platform.

Needed:

- keep v0.9 DNS tunnel behavior covered by VM regression tests
- DoH executes in v0.9.1 as optional DNS helper upstream behavior, with focused
  tests and the same upstream failure diagnostics path as direct/tunnel DNS
- add full local DNSSEC validation if upstream-AD is not enough
- keep IDNA canonicalization visible in tests
- record cache, stale, upstream, DNSSEC, and failure facts in diagnostics

Boundary:

- DNS remains an optional helper
- core transport must not depend on DNS helper availability
- no ad hoc DNS packet parsing; keep using dnspython

## Priority 5: Carrier Contract And QUIC/HTTP3 Datagram Work

Goal:

Keep the carrier layer narrow and boring while providing real standard-protocol
carrier adapters for direct QUIC DATAGRAM and HTTP/3 DATAGRAM.

Direct QUIC DATAGRAM is the first v0.9.1 carrier choice because it is the
lowest-friction second transport after UDP:

- it preserves datagram semantics
- it gives NAT/firewall behavior different from raw UDP
- it can share TLS/QUIC ecosystem tooling
- it avoids pretending Gatherlink is a browser/WebSocket product
- it gives labs a meaningful carrier-comparison target

HTTP/3 DATAGRAM is also an intended additive variant. It carries the same
Gatherlink UDP-format carrier packet inside HTTP/3 datagrams. It exists for
infrastructure compatibility and future HTTP-aware deployment paths, not because
Gatherlink needs HTTP request semantics.

Needed:

- explicit config for direct QUIC and HTTP/3 DATAGRAM carrier endpoints
  (first config slice implemented with path `carrier` values `udp`,
  `quic-datagram`, and `http3-datagram`, plus optional
  `carrier_max_datagram_size`)
- fail-closed bridge tests for both non-UDP carrier names so they cannot be
  accidentally executed by the Rust UDP runner without the Python carrier adapter
- Python carrier adapters for direct QUIC DATAGRAM and HTTP/3 DATAGRAM. These
  adapters bind local UDP sockets for Rust and move the resulting opaque packet
  bytes over standard datagram protocols.
- runner supervision for non-UDP paths: Python starts the adapter, rewrites the
  Rust-facing runtime path to local UDP endpoints, and stops the adapter with
  the service lifecycle.
- packet movement parity with UDP carrier for opaque Gatherlink frames, proven
  by byte-preserving lab smoke tests in both directions.
- Traefik UDP-forwarding smoke for QUIC DATAGRAM and HTTP/3 DATAGRAM carriers
  through `gatherlink lab carrier-proxy-smoke ... --proxy traefik`.
- docs/lab acceptance shapes that describe the required VM/proxy proofs without
  pretending external proxy labs were run when only local carrier smokes passed.

Deferred beyond the local v0.9.1 carrier smoke:

- diagnostics for connect, reconnect, path readiness, datagram send/receive,
  loss/drop, and close
- VM/lab proof that the same service can run over UDP, direct QUIC DATAGRAM,
  HTTP/3 DATAGRAM, and proxied carrier paths under impaired network conditions
- clear MTU/datagram-size handling
- broader fail-closed behavior when external QUIC, HTTP/3, auth, or proxy setup
  is invalid.

Boundary:

- both carriers carry already-formed Gatherlink UDP-format carrier packets
- it must not inspect endpoint service payloads
- it must not make routing, helper, trust, or scheduler policy decisions
- it must not change Gatherlink packet headers, encryption, replay protection,
  routing context, aggregation behavior, or service semantics
- at the sink, the carrier is terminated and unwrapped, then the recovered
  Gatherlink packet is handled exactly like a packet received from the UDP
  carrier
- HTTP/3 request/session behavior is carrier machinery only; follow
  `docs/architecture/architecture-contract.md` for carrier boundaries
- the current unwrap path is a Python-owned standard-protocol adapter around the
  Rust UDP packet executor. That is correct for v0.9.1 because it keeps Rust
  policy-free and uses the established protocol library instead of custom QUIC
  machinery.
- Rust may own packet-time carrier execution later if profiling/design justifies
  it, but it should receive compact carrier facts only

Deployment note:

QUIC can sit behind UDP-capable layer-4 forwarding/proxying when that proxy
preserves UDP datagrams to the Gatherlink sink. Examples include Traefik UDP
routers or Cloudflare Spectrum-style TCP/UDP proxying. Ordinary HTTP/3 or web
reverse proxy behavior is not enough unless it provides the required UDP/QUIC
pass-through semantics.

Direct internet exposure must remain supported for simple labs and explicit
operator choice, but it is not the recommended public deployment shape. For
internet-facing use, prefer placing Gatherlink carriers behind Cloudflare
Spectrum-style TCP/UDP protection and/or Traefik UDP forwarding where that is
available. Tests and sample configs should still include no-proxy direct
carrier examples so carrier behavior stays easy to validate without external
services.

The Traefik proof is useful v0.9.1 evidence when an operator can run it. The
v0.9.1 VM pass ran the carrier proxy smoke for both QUIC DATAGRAM
and HTTP/3 DATAGRAM through Traefik UDP forwarding. Keep setup guidance in
`docs/labs/quic-traefik-proxy.md`, and do not claim broader impaired-network or
internet proxy acceptance unless those labs have actually run.

HTTP/3 DATAGRAM is a separate carrier variant, not a replacement for direct
QUIC DATAGRAM. Ordinary HTTP/3 request proxying is still not sufficient; the
implementation must explicitly support HTTP/3 datagrams.

Keep HTTP/3 DATAGRAM acceptance guidance in
`docs/labs/http3-datagram-carrier.md`.

Future carrier candidates live in `docs/reports/future-roadmap-pipeline.md`.
- browser-specific transport behavior

Future carriers follow the same rule: TCP/TLS, WSS, and obfuscation profiles are
outer transport wrappers for the same Gatherlink UDP-format carrier packets.
They must not introduce a second Gatherlink packet model.

CONNECT-UDP/MASQUE is a good future candidate when Gatherlink needs
standards-based UDP proxying through HTTP infrastructure. It should not be
implemented in v0.9.1 unless explicitly promoted. If implemented later, it follows
the same carrier-wrapper invariant: it transports the same Gatherlink UDP-format
carrier packet and does not own routing, identity, helper, control, encryption,
or packet semantics.

## Priority 6: WireGuard Lifecycle Automation

Goal:

Keep Gatherlink as the transport coordinator while WireGuard remains
WireGuard-owned.

Needed:

- generate or validate WireGuard endpoint snippets against Gatherlink service
  mappings
- optionally supervise `wg`/`wg-quick` lifecycle through explicit helper
  commands
- diagnose missing tools, unsafe routes, and endpoint mismatch
- prove WireGuard-over-Gatherlink in the VM harness when possible

Boundary:

- Gatherlink must not parse WireGuard packets
- Gatherlink must not own WireGuard firewall/routing policy
- platform-specific interface work must go through Debian compatibility tooling

## Priority 7: Operator-Safe Lab Bundles And Guided Setup

Goal:

Make the proven multi-node lab flows boring to recreate without making
Gatherlink an unsafe host mutator. Gatherlink should generate, validate, and
explain the setup; privileged package, interface, route, and firewall changes
remain explicit operator actions unless a later command is deliberately run with
clear scope.

Needed:

- generate a lab bundle for common topologies, starting with the three-VM
  Hyper-V shape that models one sink with two source nodes and later routing
  experiments. First slice implemented by `gatherlink lab bundle
  hyperv-three-node --out ...`.
- emit per-node Gatherlink configs, WireGuard endpoint/interface plans,
  expected private-path addressing, service names, monitor commands, and
  cleanup manifests
- include a grouped monitor target, for example a generated command equivalent
  to monitoring both sink-side services in the WireGuard demo
- generate exact curl/probe commands for each source service so an operator can
  prove data moved through the intended tunnel
- add a read-only preflight/doctor mode that checks generated config validity
  and manifest completeness now. VM reachability, path IP reachability,
  required tools, Gatherlink version/commit parity, stale interfaces,
  service-port collisions, and missing private paths are next extensions once
  inventories are supplied.
- add an `--emit-debian-commands` style output for missing packages and
  interface setup, with no automatic package installs by default
- write a manifest of created services, WireGuard interfaces, temporary files,
  probe servers, ports, and diagnostics paths so cleanup can be explicit and
  scoped
- make cleanup consume the manifest rather than using broad process/interface
  matching. First slice implemented by `gatherlink lab cleanup manifest.json`,
  which renders scoped commands only.
- keep host-local values, SSH keys, host keys, generated WireGuard private keys,
  and VM management addresses out of committed examples and reports unless
  intentionally redacted

Useful command shapes to design:

```bash
gatherlink lab bundle hyperv-three-node --out ./lab-out
gatherlink lab preflight ./lab-out/manifest.json
gatherlink lab cleanup ./lab-out/manifest.json
gatherlink helpers wireguard-plan --bundle ./lab-out/manifest.json
gatherlink services monitor --group wg-demo
```

Boundary:

- Gatherlink may generate plans, validate state, start Gatherlink-managed
  services, emit explicit privileged commands, and clean up objects recorded in
  its manifest
- follow `docs/architecture/architecture-contract.md` and
  `docs/helpers/wireguard-helper.md` for host mutation and helper boundaries
- VM/Hyper-V specifics stay in lab tooling and docs, not core runtime

## Priority 8: Multi-Hop Relay Policy Automation

Goal:

Take the existing secure relay-hop executor and runner toward repeatable
multi-hop policy.

Needed:

- signed topology/control input for multi-hop relay plans
- next-hop authorization checks across more than one hop
- loop/hop-limit protection
- diagnostics explaining selected and rejected relay paths
- VM/lab proof for at least one multi-hop route before marking this complete

Boundary:

- follow `docs/protocol/relay-session-lifecycle.md`,
  `docs/protocol/relay-trust-model.md`, and
  `docs/architecture/architecture-contract.md`

## Priority 9: Experimental REST Writes, If Still Wanted

Goal:

Decide whether the experimental local REST helper should gain selected write
APIs or remain read-only.

Allowed if implemented:

- loopback by default
- explicit CLI startup
- one-hour write window by default
- write APIs map to existing CLI-equivalent operations
  (first supported endpoint: `POST /v1/services/{name}/close`)
- no secret material in responses
- tests for write expiry and fail-closed behavior

Not allowed:

- remote unauthenticated management
- long-lived write access
- separate policy behavior that diverges from CLI/runtime state

## Priority 10: Diagnostics Polish

Goal:

Make diagnostics useful enough that VM and soak failures are quick to explain.

Needed:

- scenario-level helper events where coverage is still thin
- QUIC carrier events and carrier comparison summaries
- relay drop and rekey event summaries in reports
- redaction checks in new status/report outputs
- stable event codes for any v0.9.1 feature
- JSONL validation in all acceptance reports

## Priority 11: Packaging And Operator Fit

Goal:

Make Debian-only operation feel intentional instead of incidental, and make
v0.9.1 installable from GitHub release artifacts without requiring developers to
build from source.

Needed:

- GitHub release artifacts for Debian users
- at least one boring package/archive format that installs the Python CLI,
  Rust operator helper binary, helper entry points, docs, and example configs
- checksum files for release artifacts
- first artifact tooling exists at `tools/release/prepare_artifacts.py`; it
  builds a tracked source archive, local no-dependency Python wheel, the
  Debian-oriented `gatherlink-time-helper` Rust binary, checksum file, and
  `docs/user/` Wiki payload. Generated artifacts stay out of git and are
  recreated for releases.
- release notes that state Debian-only support and list tested lab/VM gates.
  First slice is `docs/releases/v0.9.1.md`.
- clearer install notes for fresh installs and upgrades from previous
  development checkouts
- automated release/package tooling that publishes or prepares the short
  `docs/user/` pages for the GitHub Wiki, while keeping repository docs
  canonical
- service unit examples if systemd management becomes common
- package/runtime path checks in `gatherlink doctor`. First release-artifact
  check validates source archive, Python wheel, Rust helper binary, checksums,
  and Wiki payload shape.
- migration notes for state/config layout changes
- release checklist tied to tests, VM reports, and docs
- sample configs that work without Cloudflare, Traefik, or other external
  services

Deferred to the future pipeline:

- signed package repository/channel
- automatic update agent
- automatic rollback tooling
- broad appliance-style update orchestration
- non-Debian package formats

Boundary:

- v0.9.1 packaging is about reproducible GitHub release artifacts and clear
  operator install/upgrade notes
- GitHub Wiki pages are release documentation copies, not canonical specs
- Wiki publishing must be part of the package/release workflow; it must not
  rely on someone remembering a manual copy step
- package scripts must not hide policy, provisioning, or topology behavior
- package layout must preserve the Debian compatibility boundary
- release artifacts must not contain secrets or environment-specific VM config

## Code Hygiene Rule For V0.9.1

Every v0.9.1 feature should use the same three-pass rule as v0.9:

1. Make it work with tests and docs.
2. Review placement, boundaries, names, and failure behavior.
3. Re-review, simplify, rerun tests/labs, then commit the coherent slice.

Any new package or module must have real behavior, tests, and a documented
owner. Deferred ideas stay in docs.
