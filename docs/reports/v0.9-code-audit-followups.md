## V0.9 Boundary Finding

### Production Discovery And Remote Status Must Not Be Lab-Only

Production must own sparse discovery/control metadata announcements,
monitor-triggered control cadence, and explicit temporary remote-status
request/response handling.

Current audit status:

- `announce_control_metadata()` is wired into the normal Rust-backed core
  runner at sparse baseline cadence, with monitor cadence requested over
  production service IPC.
- Remote-status request/response payloads are decoded in
  `python/gatherlink/control/remote_status.py`, and the production runner
  handles reserved service id 8 through the shared reserved-service dispatcher.
- Shared-sink remote-status replies use the authenticated peer/session scope
  attached to the reserved-service event by Rust. Python owns the message
  semantics and asks Rust only to send the already-encoded service payload back
  to that authenticated peer scope; Rust does not decode remote-status JSON.
- Normal core service IPC exposes `control-cadence` and `remote-status`
  commands. Lab services use the same command/message shape for accelerated
  checks.
- `services list` can show learned remote service metadata as read-only remote
  entries once discovery has arrived, and `services monitor` requests temporary
  remote status while an operator is watching.
- Endpoint assertions remain recorded verification facts. They are not compared
  against the local service target automatically because shared-sink and
  peer-scoped-source services intentionally have different local app endpoints
  on each side. A future explicit expected-peer-endpoint config field can turn
  those facts into fail-closed policy without making Python infer topology.

Closure requirement:

- labs may enable, accelerate, or assert discovery/status behavior, but the
  implementation must live in production-owned control/runtime modules;
- lab services and normal services should use the same IPC command shape for
  monitor cadence and the same reserved-service decoder path for remote status;
- v0.9 tests must prove the production path, not a lab-only implementation.

Verification now completed:

- local hidden-sink lab mode proves remote status can be requested through the
  production reserved-service lane without reading the sink's local IPC socket;
- the three-WSL shared-sink check proves two authenticated sources can share one
  sink carrier port, exchange UDP payloads, and receive remote sink status rows
  over the same production remote-status module.

## Fixed In Current Code

- DNS Gatherlink-tunnel upstream execution is implemented in the Python DNS
  helper. It sends DNS wire queries to an explicitly configured local
  Gatherlink UDP service endpoint; Rust remains unaware of DNS semantics.
- Process-managed DNS helper launches receive the shared `--diagnostics-jsonl`
  flag through the helper supervisor.
- DNS helper focused tests cover direct upstreams, tunnel upstream policy,
  actual UDP datagram exchange with a service endpoint, DoH fail-closed
  behavior, CLI parsing, diagnostics, and supervisor wiring.
