# V0.9.3 Roadmap

Status: release candidate on `dev`; ready for final soak/tag decision.

V0.9.3 is an adaptive performance and real-world operation release. V0.9.2 is
the stable baseline: Debian-first, CLI-first, encrypted multipath UDP
transport, Python-owned policy, and Rust-owned packet execution. V0.9.3 should
make that system smarter under real network pressure without broadening the
product into a mesh, GUI, hosted control plane, generic proxy product, or
non-Debian platform project.

## Theme

Make Gatherlink choose better automatically.

V0.9.3 should improve:

- benchmark and profiler discipline
- coordinated adaptive scheduling
- ordered multipath for TCP-like UDP services
- per-service scheduling and outcome feedback
- TCP-aware helper paths where WireGuard-over-Gatherlink cannot expose stream
  meaning
- autonomous secure-session rekeying
- congestion fairness beyond local credits
- release evidence that stays easy to compare against v0.9.2

Living release notes are maintained in [`docs/releases/v0.9.3.md`](../releases/v0.9.3.md). Every
operator-visible slice should update those notes when it changes config shape,
CLI behavior, service registry behavior, helper contracts, diagnostics schema,
runtime JSON, compatibility, or release-gate evidence.

## Standing Rules

- Follow [`docs/operations/development-discipline.md`](../operations/development-discipline.md) and
  [`docs/operations/release-development-process.md`](../operations/release-development-process.md).
- Keep Python responsible for config, policy, orchestration, helpers,
  diagnostics, provisioning, scheduler intelligence, and operator explanations.
- Keep Rust responsible for compact packet execution, sockets, counters, AEAD,
  replay, batching, fragmentation, bounded queues, dedupe, and cheap compiled
  scheduling primitives.
- Do not add hidden retransmission, reinjection, or reliability semantics for
  ordinary application UDP payloads.
- Do not inspect encrypted helper payloads, including WireGuard payloads, to
  infer TCP streams.
- Do not add packet-header fields for scheduler policy unless every compact
  alternative has been exhausted and documented.
- Keep every roadmap row status-explicit: `planned`, `implemented`,
  `unit-tested`, `lab-tested`, `VM-tested`, or `closed`.
- Commit coherent slices with docs and tests close to the code they prove.

## Priority 2: Benchmark And Profiler Discipline

Goal:

Make benchmark evidence easier to run, compare, and trust.

Work:

- maintain a current-results table and a historical-results table
- every performance row should show the test date, profile, scheduler, traffic
  shape, path speeds, MTU, offered load, delivered load, and comparison
  percentages where applicable
- comparison percentages should include direct kernel WireGuard, userspace
  WireGuard, raw Gatherlink, and the current recommended automatic scheduler
  when those baselines exist
- keep wireguard-go as the primary userspace WireGuard comparison baseline
  until repeated evidence says otherwise
- keep optional GotaTun and BoringTun rows as backend comparison evidence, not
  Gatherlink dependencies
- add or tighten benchmark tooling that records CPU, Linux UDP counters,
  datagrams per second, drops, queue pressure, retransmits, and path split
- add path profiling/export tooling that can turn real-link observations into
  repeatable lab shapes. Status: implemented, unit-tested, and proven against a
  real VM status snapshot. `tools/export_observed_status_profile.py` exported
  `.gatherlink/profile-export-proof/current/profile.json` from
  `.gatherlink/hyperv-performance/20260526T142452Z-v093-coordinated-acceptance/status-b.json`,
  preserving observed per-path counters, delay, jitter, loss, MTU, and pressure
  into a repeatable profile draft.
- keep benchmark control metadata bounded under pressure. Status: implemented
  for opportunistic data-timing samples; local profile proof reruns cleanly
  after exposing a 1200-MTU control-frame overflow before VM proof.

Acceptance:

- benchmark docs explain exactly what each percentage is relative to
- a focused benchmark unit/schema test proves the row format does not regress
- at least one local lab or VM benchmark run writes the current row format
- docs state when a row is current, historical, optional, or environment
  dependent

Status: implemented, unit-tested, local-lab-tested for schema/report/export
tooling, and VM-smoke-tested for current row capture and real-observation
profile export.

## Priority 3: Coordinated Adaptive Scheduler V2

Goal:

Improve the default automatic scheduler so normal operators do not need to
choose a specialist scheduler for common real-world profiles.

Work:

- improve policy selection for TCP-over-WireGuard-over-Gatherlink, mixed
  TCP+UDP dual-WireGuard, fiber+5G, Starlink+5G, five-path Starlink-like, lossy,
  jittery, and asymmetric profiles
- add confidence-scored path responsiveness classes such as `wired_stable`,
  `cellular_like`, `starlink_like`, `volatile_wireless`, and `unknown`.
  Status: implemented and unit-tested as weak Python-owned coordinator
  diagnostic hints with bounded per-path history exported in
  `scheduler.decision` path profiles. VM tuning has current proof rows for
  `acceptance-300-500-700` and `realworld-starlink-plus-5g` under
  `.gatherlink/hyperv-performance/20260526T142452Z-v093-coordinated-acceptance/`
  and
  `.gatherlink/hyperv-performance/20260526T142614Z-v093-coordinated-starlink/`.
- use fixed operator config as the initial truth and override inference only
  when telemetry is sustained and credible
- keep flapping protection based on both packet volume and a short time guard
- emit concise diagnostics explaining mode changes, rejected changes, and
  fallback decisions

Acceptance:

- unit tests for class selection, dwell guards, confidence, and fallback
- benchmark rows comparing coordinated adaptive beside specialist schedulers
- no regression in raw Gatherlink high-speed and representative real-world
  profiles

Status: implemented, unit-tested, and VM-smoke-tested for the v0.9.3 release
scope. Benchmark-driven tuning remains the normal development method for future
releases, but no v0.9.3 scheduler-proof item is left pending here.

## Priority 4: Ordered Multipath For TCP-Like UDP

Goal:

Make ordered multipath safer and more useful for opaque ordered UDP services,
especially WireGuard carrying TCP, without claiming hidden reliability.

Work:

- use receiver feedback, bounded in-flight work, latency confidence, reorder
  pressure, and sender-side pacing to avoid damaging TCP-like flows. Status:
  implemented and unit-tested for p95 latency in ordered virtual timelines and
  receiver reorder-buffer depth/age credit tightening. VM contrast proof at
  `.gatherlink/hyperv-performance/20260526T142533Z-v093-ordered-starlink/`
  confirms ordered mode is still specialist/research-only for the Starlink+5G
  shape, not an automatic default.
- improve earliest-arrival estimates from path latency, queued/in-flight work,
  path capacity, and receiver pressure
- keep `ordered_multipath_capacity_aware` separate from `ordered_multipath` so
  experiments remain comparable
- explicitly document when ordered mode is specialist/research versus automatic
  default

Acceptance:

- unit tests for credit calculation, receiver pressure, reorder hold, and
  rejected/impossible telemetry
- benchmark proof that ordered mode improves at least one named profile or stays
  safely below the automatic default
- no hidden retry/reinjection for ordinary UDP payloads

Status: implemented for the v0.9.3 specialist/research scope, unit-tested, and
VM-contrasted. Ordered mode is intentionally not promoted to the default because
current VM evidence still trails coordinated adaptive on WAN-like shapes. Future
work may improve it, but v0.9.3 closes by documenting the safe boundary rather
than pretending ordered multipath is ready as an automatic TCP policy.

## Priority 5: Service-Aware Scheduling And Outcome Feedback

Goal:

Let each service get a suitable path policy while still respecting total path
pressure.

Work:

- polish per-service policy compilation for `tcp_ordered`, `udp_bulk`,
  `latency_sensitive`, `control`, and `unknown`. Status: implemented and
  unit-tested for same-pass bulk capacity reservation so several bulk services
  do not all assume the same spare path is empty.
- improve service budget/QoS so TCP-stable services can be protected without
  starving UDP-fast services. Status: implemented and unit-tested for runner
  `service_budget` status, including active state, reason, packet/byte
  overrides, and compact service-rate samples for monitor output.
- make dual-WireGuard advanced mode easier to benchmark and explain
- consume outcome facts such as TCP retransmits or helper delivery failure when
  they are available, without requiring them for normal operation
- keep fixed service splits as advanced controls; the general target remains
  good automatic behavior

Acceptance:

- unit tests for per-service policy reconciliation and starvation prevention
- mixed TCP+UDP benchmark rows for automatic and advanced dual-WireGuard modes
- diagnostics explain service allocation, protected-service choices, and
  bulk-service expansion/shrink decisions

Status: implemented for the v0.9.3 release scope and unit-tested with
operator-visible service budget status. VM helper acceptance covers explicit
SOCKS5/TCP-forward service paths, and benchmark rows cover current mixed
TCP+UDP behavior. Future releases can continue tuning mixed-service policy, but
the v0.9.3 service-aware scheduling proof is complete.

## Priority 6: TCP-Aware Proxy Plus WireGuard Hybrid

Goal:

Give TCP-heavy users a path that exposes TCP stream meaning to Python helpers
without inspecting encrypted WireGuard packets or moving TCP semantics into
Rust.

Work:

- implement explicit TCP proxy mode first with configured listen endpoints
- implement or harden the companion exit helper that connects to the configured
  destination
- report stream ids, open/close/reset, backlog, credit, and traffic-class hints
  as Python/helper facts. Status: implemented for explicit helper mode and
  unit-tested for bounded helper stream `traffic_class` and `flowlet_key`
  open-frame hints, credit events, reset events, and close outcome facts with
  byte/frame/backlog counters.
- prove explicit SOCKS5 over Gatherlink locally. Status: lab-tested with
  `tools/socks5_gatherlink_acceptance.py`, which starts two real core services,
  the SOCKS5 helper, companion stream exit, and status HTTP helper, then fetches
  the HTTP page through the Gatherlink UDP stream service.
- keep WireGuard-over-Gatherlink as the default path for non-TCP traffic
- keep transparent Debian TPROXY/nftables interception as a separate opt-in
  helper milestone only after explicit mode is proven

Acceptance:

- explicit-mode unit and helper smoke tests
- lab or VM proof against a simple HTTP service over Gatherlink
- benchmark comparison against WireGuard-over-Gatherlink and direct WireGuard
  baselines on clean, fiber+5G, Starlink+5G, and LTE-style profiles
- cleanup/revert tests for any Gatherlink-labeled firewall rules when the
  transparent mode is promoted

Status: implemented enough for v0.9.3 explicit-helper proof: unit-tested,
local-lab-tested, and VM-tested for SOCKS5 CONNECT and TCP-forward over
Gatherlink against the status HTTP helper at
`.gatherlink/hyperv-socks5-acceptance/20260526T073135Z/`. Local explicit
SOCKS5-over-Gatherlink correctness was re-proven at
`.gatherlink/socks5-gatherlink-acceptance/v093-roadmap-proof/`, including
helper stream open/credit/close diagnostics. VM throughput benchmarking is now
proven for the explicit TCP-forward helper at
`.gatherlink/hyperv-socks5-acceptance/20260526T-v093-tcp-helper-throughput-300m-active/`:
an 8-second 300 Mbit/s capped TCP stream delivered at 300.14 Mbit/s active sink
rate with zero byte delta. This is a capped helper-throughput proof, not a
claim that explicit TCP helpers beat all WireGuard-over-Gatherlink profiles.

## Priority 7: Autonomous Rekey And Handshake Hardening

Goal:

Finish the secure-session story that v0.9.2 prepared but did not fully claim.

Work:

- complete autonomous peer-to-peer live rekey while traffic is moving
- add an in-band reserved auth/crypto rekey payload that carries Noise IK
  initiation/response facts without traffic keys. Status: implemented,
  unit-tested for request/response/reject payloads, receiver-index freshness,
  topology freshness, inverse replacement sessions, and VM-tested for a
  single-initiator live rekey while traffic is moving.
- preserve keyless authenticated-session metadata in Noise-generated security
  blocks. Status: implemented and unit-tested for config/runtime expansion and
  runtime reconstruction of the current authenticated session; older
  authenticated blocks without metadata remain packet-execution compatible but
  cannot originate autonomous live rekey and now produce a runtime-plan warning.
- route reserved auth/crypto service payloads from Rust to Python in the
  foreground runner. Status: implemented and unit-tested for invalid payload
  local `rekey.rejected` diagnostics and structured decoded results;
  accepted-session hot apply is implemented and unit-tested in Python runtime
  helpers; decoded valid messages are kept as operator-safe bounded IPC/status
  facts. Production and lab runners explicitly apply the reserved auth/crypto
  service scheduler as a Python-owned primitive while keeping Noise/retry
  semantics out of Rust. The Python `LiveRekeyCoordinator` is implemented and
  unit-tested for duplicate start suppression, valid request/response
  hot-apply, and bounded reject payloads. Foreground `run service` now accepts
  explicit rekey identity/topology/trust-root inputs, evaluates rekey cadence in
  Python, transmits reserved auth/crypto payloads through Rust unchanged, and
  hot-reapplies validated replacement security DTOs to Rust. VM proof exposed
  and fixed two real integration issues: live rekey now reads the real
  `service_stats` counter shape, and responders transmit rekey responses before
  hot-applying replacement AEAD so initiators can decrypt the response under the
  current session. Passing evidence:
  `.gatherlink/hyperv-rekey-acceptance/20260526T080242Z/`.
- expose operator-safe auth/crypto status in service monitor. Status:
  implemented and unit-tested for compact `auth/rekey control` rows that show
  message type, peer, topology generation, receiver index, path, sequence, and
  whether a Noise payload was present without exposing Noise bytes or keys.
- keep receiver-index clarity and replay reset behavior explicit
- preserve overlap windows only as long as needed
- add cookie/anti-DoS behavior only if it can be scoped cleanly
- keep invalid secure packets silent on the network and counted locally

Acceptance:

- unit tests for success, expiry, stale topology, peer mismatch, replay reset,
  overlap windows, and invalid packet drops
- lab/VM proof of live rekey while traffic is moving
- diagnostics for start, success, reject, expiry, and fallback paths

Status: implemented, unit-tested, and VM-tested for the documented
single-initiator live-rekey path. Simultaneous/colliding rekey initiation is
not yet claimed and should remain a later hardening scenario.

## Priority 8: Congestion Fairness Beyond Local Credits

Goal:

Make Gatherlink a better network citizen on shared, volatile, or bufferbloaty
links without turning application UDP into reliable transport.

Work:

- add explicit self-limiting behavior when path pressure suggests Gatherlink is
  competing badly with other traffic. Status: implemented and unit-tested for
  Python-compiled inferred pacing budgets, with explicit config override and
  `off` bypass preserved.
- add responsiveness policies such as `conservative`, `adaptive`, and
  `volatile`. Status: implemented as `scheduler.congestion_policy`, where
  Python chooses the backoff factor and Rust only executes a primitive
  `pacing_budget_bps`.
- use path-local queue growth, drops, latency-under-load, and recovery behavior
  to choose compiled limits
- emit operator diagnostics explaining when Gatherlink backs off or recovers.
  Status: implemented and unit-tested for `scheduler.decision` details that
  include congestion policy, pressure level, reason, capacity, and compiled
  `pacing_budget_bps`; recovery hysteresis is implemented and unit-tested in
  Python reapply. VM proof now runs direct path-level competing UDP pressure
  beside Gatherlink traffic so the self-limiting work can be tested against
  real shared-link pressure, not only deterministic lab reports.

Acceptance:

- repeatable tests with competing traffic. Status: implemented, unit-tested for
  the deterministic lab-report congestion matrix, and VM-tested with
  `.gatherlink/hyperv-performance/20260526T104703Z-v093-competing-traffic-proof/`.
  That proof carried 180 Mbit/s of Gatherlink UDP while three direct path
  competitors each sent and received 80 Mbit/s with zero packet delta.
- no hidden reliability for ordinary UDP payloads
- diagnostics explain self-limiting and recovery
- clean/wired profiles do not flap when volatile policies are not selected

Status: implemented, unit-tested, and VM-tested for the current v0.9.3 scope.
The 2026-05-26 managed VM acceptance rerun proves scheduler reapply, path
fail/recover, diagnostics, and asymmetric shaping still work; the competing
traffic VM proof proves direct path pressure can run beside Gatherlink traffic
without losing either stream in the tested acceptance profile.
